# 88. 개인정보, 규제, 데이터 거버넌스

> 개발자는 기능 구현뿐 아니라 어떤 데이터를 왜 수집하고 어디에 저장하며 언제 삭제하는지 설명할 수 있어야 합니다.

## 1. 개인정보 사고방식

개인정보는 “DB 컬럼”이 아니라 사용자 권리와 조직 리스크입니다.

기본 원칙:

- 목적 제한.
- 최소 수집.
- 보관 기간 제한.
- 접근 최소화.
- 암호화와 마스킹.
- 파기 가능성.
- 감사 가능성.

## 2. 데이터 분류

| 분류 | 예시 | 처리 |
|---|---|---|
| 공개 | 공개 문서 | 일반 관리 |
| 내부 | 운영 지표 | 접근 제어 |
| 개인정보 | 이름, 전화번호, 이메일 | 암호화/마스킹/권한 |
| 민감정보 | 금융정보, 신분증, 건강정보 | 강한 통제와 감사 |
| 비밀 | API key, token | secret manager |

데이터 분류가 없으면 보호 수준을 정할 수 없습니다.

## 3. 수집과 동의

- 수집 목적을 명확히 한다.
- 필수/선택 항목을 구분한다.
- 동의 철회 경로를 제공한다.
- 이벤트 로그에도 개인정보가 들어갈 수 있음을 고려한다.
- third-party SDK가 무엇을 수집하는지 확인한다.

## 4. 보관과 파기

- 보관 기간을 정의한다.
- soft delete와 hard delete를 구분한다.
- 백업 데이터의 파기 정책을 고려한다.
- 분석 데이터는 익명화/가명화한다.
- 법적 보관 의무와 사용자 삭제 요청의 충돌을 문서화한다.

## 5. 접근 통제와 감사

- role 기반 접근.
- break-glass 절차.
- 관리자 조회 감사 로그.
- 대량 다운로드 탐지.
- 운영 DB 직접 접근 최소화.
- 개인정보 조회 사유 기록.

## 6. 국경 간 데이터와 외부 위탁

글로벌 서비스는 데이터가 어느 국가에 저장되고 처리되는지 중요합니다.

- data residency.
- cross-border transfer.
- subprocessors.
- DPA.
- 위탁사 보안 평가.

개발자는 외부 SaaS와 로그 수집 도구도 데이터 이전 경로임을 알아야 합니다.

## 7. Privacy by Design

설계 단계에서 확인할 질문:

- 이 데이터를 꼭 수집해야 하는가?
- 덜 민감한 대체 데이터가 있는가?
- 익명화해도 목적을 달성하는가?
- 권한 없는 개발자가 볼 수 있는가?
- 로그와 백업에도 남는가?
- 삭제 요청 시 어디까지 삭제되는가?

## 8. 체크리스트

- [ ] 데이터 분류표가 있다.
- [ ] 개인정보가 로그/이벤트/분석 도구에 노출되지 않는다.
- [ ] 보관 기간과 파기 정책이 구현되어 있다.
- [ ] 관리자 접근 감사 로그가 있다.
- [ ] 외부 위탁/SDK 데이터 흐름을 파악한다.
- [ ] 삭제 요청과 백업 보관의 정책이 명확하다.
