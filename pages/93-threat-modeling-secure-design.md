# 93. 위협 모델링과 보안 설계

> 보안은 취약점 스캔만으로 해결되지 않습니다. 설계 단계에서 공격자를 상상해야 합니다.

## 1. 위협 모델링 질문

- 보호해야 할 자산은 무엇인가?
- 공격자는 누구인가?
- 진입점은 어디인가?
- 신뢰 경계는 어디인가?
- 실패하면 어떤 피해가 나는가?
- 어떤 통제로 위험을 줄일 수 있는가?

## 2. STRIDE

| 분류 | 의미 | 예 |
|---|---|---|
| Spoofing | 신원 위조 | 세션 탈취 |
| Tampering | 변조 | 요청 금액 변경 |
| Repudiation | 부인 | 감사 로그 없음 |
| Information Disclosure | 정보 노출 | 로그에 토큰 출력 |
| Denial of Service | 서비스 거부 | 비싼 요청 반복 |
| Elevation of Privilege | 권한 상승 | 관리자 API 접근 |

## 3. 신뢰 경계

신뢰 경계는 데이터가 다른 신뢰 수준으로 이동하는 지점입니다.

- 브라우저 → 서버.
- API Gateway → 내부 서비스.
- 서비스 → DB.
- 운영자 → 관리자 도구.
- 외부 webhook → 내부 큐.

경계마다 인증, 인가, 검증, 로깅이 필요합니다.

## 4. Attack Tree

공격 목표를 트리로 쪼갭니다.

예: 계좌 잔고 탈취

- 로그인 탈취.
- 세션 재사용.
- 권한 검증 우회.
- 주문 API 파라미터 변조.
- 관리자 도구 오남용.

각 가지에 탐지와 방어를 붙입니다.

## 5. 보안 설계 리뷰

보안 리뷰에서 볼 것:

- 인증/인가 위치.
- 입력 검증.
- secret 흐름.
- 민감정보 저장/로그.
- rate limit.
- audit log.
- 관리자 기능.
- 외부 연동 실패 모드.

## 6. 체크리스트

- [ ] 자산과 공격자를 정의했다.
- [ ] 데이터 흐름도와 신뢰 경계가 있다.
- [ ] STRIDE로 주요 위협을 훑었다.
- [ ] 관리자/운영자 경로도 검토했다.
- [ ] 탐지와 감사 로그가 있다.
- [ ] 잔여 위험을 문서화했다.
