# 69. 알고리즘 문제 해결과 실무 적용

> 알고리즘은 코딩테스트용 퍼즐이 아니라 데이터가 커져도 시스템이 버티게 하는 사고방식입니다.

## 1. 문제 해결 순서

1. 입력과 출력 정의.
2. 입력 크기와 제약 확인.
3. 단순한 방법부터 생각.
4. 병목을 찾고 자료구조로 줄임.
5. edge case를 테스트.
6. 복잡도와 실제 비용을 설명.

## 2. 복잡도 함정

- O(1)도 네트워크 호출이면 느리다.
- O(log n)도 디스크 I/O가 섞이면 비용이 크다.
- O(n)은 n이 작으면 가장 단순하고 빠를 수 있다.
- O(n²)은 작은 데이터에서 숨어 있다가 운영 데이터에서 터진다.

## 3. 정렬

정렬은 검색, 중복 제거, 그룹핑, top-k 문제의 기본입니다.

- stable sort: 같은 키의 기존 순서 유지.
- partial sort/top-k: 전체 정렬보다 heap이 유리할 수 있다.
- external sort: 메모리에 다 안 올라가는 데이터를 디스크 기반으로 정렬.
- DB에서는 `ORDER BY`가 인덱스를 타는지 확인해야 한다.

## 4. 탐색

### Binary Search

정렬된 공간에서 답을 좁힙니다.

- 숫자 범위 최적화.
- lower_bound/upper_bound.
- 시간표, 가격 구간, 버전 탐색.

### BFS/DFS

- BFS: 최단 거리, 단계별 확산.
- DFS: 전체 경로 탐색, 위상 정렬, 사이클 감지.

실무 예시:

- 권한 상속 그래프.
- 의존성 설치 순서.
- 추천/관계 탐색.
- 네트워크 라우팅 분석.

## 5. 해시와 집계

해시는 실무에서 가장 많이 쓰는 최적화입니다.

- 중복 제거.
- 카운팅.
- 조인 전 작은 테이블 메모리 적재.
- 캐시 키.
- idempotency key 저장.

주의:

- key 설계가 틀리면 캐시 오염이 생긴다.
- hash collision은 보안 공격 지점이 될 수 있다.
- mutable 객체를 key로 쓰면 위험하다.

## 6. 투 포인터와 슬라이딩 윈도우

정렬된 배열이나 연속 구간 문제에 강합니다.

- 연속 기간 거래량 합.
- 최근 N분 요청 수.
- 문자열 부분 구간.
- 가격 범위 매칭.

## 7. 그래프 알고리즘

| 알고리즘 | 용도 |
|---|---|
| 위상 정렬 | 의존성 순서 결정 |
| Dijkstra | 양수 가중치 최단 경로 |
| Union-Find | 그룹 연결성 판단 |
| SCC | 강하게 연결된 컴포넌트 |
| Max Flow | 용량 제한 배분 |

실무에서 그래프는 생각보다 자주 등장합니다. 조직도, 권한, 네트워크, 의존성, 추천, 결제 흐름은 모두 그래프입니다.

## 8. 동적 계획법

중복되는 하위 문제를 저장합니다.

- 할인 조합 최적화.
- 경로 비용 최적화.
- 배치 스케줄링.
- 리소스 할당.

하지만 DP는 유지보수성이 떨어질 수 있으므로 명확한 수식과 테스트가 필요합니다.

## 9. 확률적 자료구조

- Bloom Filter: 있을 수 있음/절대 없음.
- Count-Min Sketch: 근사 빈도.
- HyperLogLog: 대략적인 distinct count.

대규모 데이터에서 정확도 일부를 포기하고 메모리를 크게 줄입니다.

## 10. 실무 알고리즘 체크리스트

- [ ] 입력 크기의 상한과 평균/피크 분포를 안다.
- [ ] 단순 구현과 최적화 구현의 trade-off를 설명할 수 있다.
- [ ] 자료구조 선택 이유가 조회/삽입/정렬/범위 조건과 연결된다.
- [ ] edge case와 빈 입력을 테스트했다.
- [ ] DB/네트워크/락 비용까지 포함해 설명한다.
