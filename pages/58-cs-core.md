# 58. 개발자 CS 핵심 총정리

> 백엔드·프론트엔드·인프라 어디를 가도 통하는 기본기입니다. 특정 언어나 프레임워크보다 오래 살아남는 지식에 집중합니다.

## 한 장 요약

| 영역 | 반드시 알아야 하는 질문 |
|---|---|
| 자료구조 | 이 데이터는 어떻게 저장되고 탐색되는가? |
| 알고리즘 | 입력이 커져도 시간·메모리가 버티는가? |
| 운영체제 | 프로세스, 스레드, 메모리, 파일, 네트워크 I/O가 어떻게 동작하는가? |
| 네트워크 | 요청이 내 컴퓨터에서 서버까지 어떤 경로로 가는가? |
| 데이터베이스 | 트랜잭션, 인덱스, 락, 복구는 어떻게 일관성을 지키는가? |
| 보안 | 신뢰할 수 없는 입력·네트워크·사용자를 어떻게 다루는가? |
| 분산 시스템 | 장애가 정상인 환경에서 어떻게 일관성과 가용성을 맞추는가? |

## 1. 계산 복잡도

### Big-O 감각

- **O(1)**: 해시 조회, 배열 인덱스 접근. 단, 해시 충돌·캐시 미스·락 경합은 현실 비용을 만든다.
- **O(log n)**: 이진 탐색, B-Tree 인덱스 탐색. 데이터가 2배가 되어도 단계가 1 정도만 늘어난다.
- **O(n)**: 전체 스캔. 작은 테이블에서는 괜찮지만 요청마다 반복되면 병목이 된다.
- **O(n log n)**: 일반적인 정렬. 페이징·랭킹·배치 처리에서 자주 만난다.
- **O(n²)**: 중첩 루프. 데이터 10배면 100배 느려진다. 실무 장애의 흔한 원인이다.

### 실무 판단법

1. 입력 크기 `n`이 무엇인지 먼저 정의한다. 사용자 수, 주문 수, DOM 노드 수, 로그 라인 수가 모두 `n`이 될 수 있다.
2. 평균이 아니라 **상위 1% 입력**을 기준으로 본다.
3. CPU 시간뿐 아니라 메모리, 네트워크 왕복, DB I/O를 함께 본다.
4. 복잡한 알고리즘보다 데이터 모델·인덱스·캐시 한 줄이 더 큰 효과를 낼 때가 많다.

## 2. 자료구조

| 자료구조 | 강점 | 약점 | 실무 예시 |
|---|---|---|---|
| Array/List | 순차 접근, 캐시 친화적 | 중간 삽입/삭제 비용 | 응답 목록, 버퍼 |
| Linked List | 중간 삽입/삭제 | 탐색 느림, 캐시 비친화 | 런타임 내부 큐 일부 |
| HashMap | 평균 O(1) 조회 | 순서 없음, 해시 품질 의존 | 세션, 캐시, 중복 제거 |
| TreeMap/B-Tree | 정렬·범위 조회 | 해시보다 단건 조회 느림 | DB 인덱스, 호가 가격대 |
| Heap/PriorityQueue | 최댓값/최솟값 빠름 | 임의 삭제 어려움 | 스케줄러, 우선순위 작업 |
| Queue/Deque | FIFO/LIFO | 임의 조회 부적합 | 메시지 처리, BFS |
| Bloom Filter | 메모리 효율적 존재성 검사 | false positive 가능 | 캐시 미스 방지, 중복 후보 제거 |

### 선택 기준

- **조회가 많은가, 삽입이 많은가, 정렬이 필요한가, 범위 조건이 있는가**를 먼저 묻는다.
- HashMap을 쓰더라도 key의 동등성(`equals/hashCode`)이 안정적인지 확인한다.
- 정렬이 필요하면 매번 sort하지 말고 정렬된 구조 또는 DB 인덱스를 고려한다.

## 3. 알고리즘 패턴

### 자주 쓰는 패턴

- **정렬 + 투 포인터**: 중복 제거, 구간 합, 매칭 후보 축소.
- **해시 기반 카운팅**: 집계, 빈도 분석, 캐시 키 설계.
- **그래프 탐색(BFS/DFS)**: 의존성 그래프, 권한 상속, 라우팅, 추천.
- **동적 계획법**: 최적화 문제. 실무에서는 요금·할인·배치 최적화에 가끔 등장한다.
- **분할 정복**: 대량 처리 병렬화, map-reduce 사고방식.
- **백프레셔**: 알고리즘이라기보다 시스템 패턴. 생산 속도가 소비 속도를 넘지 않도록 제어한다.

### 개발자 면접보다 중요한 실무 버전

- 정답 코드보다 **데이터가 커질 때 실패하는 지점**을 설명할 수 있어야 한다.
- 시간 복잡도만 말하지 말고 **DB 쿼리 수, 네트워크 왕복 수, 락 대기 시간**으로 번역해야 한다.
- 캐시를 붙이면 알고리즘이 빨라지는 것이 아니라 **일관성 문제가 추가**된다.

## 4. 컴퓨터 구조와 메모리

### CPU와 메모리 계층

```
레지스터 → L1/L2/L3 캐시 → RAM → SSD/NVMe → 네트워크 스토리지
빠름/작음/비쌈                         느림/큼/상대적으로 저렴
```

- 연속 배열은 CPU 캐시에 유리하다.
- 포인터가 많은 객체 그래프는 캐시 미스를 늘릴 수 있다.
- 메모리 할당이 많으면 GC 또는 allocator 비용이 증가한다.

### 숫자와 인코딩

- 돈은 `Float/Double`이 아니라 정수 최소 단위 또는 `BigDecimal`로 다룬다.
- 문자열은 UTF-8을 기본으로 생각하되, byte 길이와 문자 길이는 다르다.
- 시간은 타임존·서머타임·영업일 캘린더 때문에 단순 덧셈으로 처리하면 위험하다.

## 5. 운영체제 핵심

### 프로세스와 스레드

- **프로세스**: 독립 주소 공간을 가진 실행 단위.
- **스레드**: 같은 프로세스의 메모리를 공유하는 실행 흐름.
- **컨텍스트 스위칭**: CPU가 실행 대상을 바꾸는 비용. 스레드를 무한히 늘리면 느려진다.

### 메모리

- **스택**: 함수 호출, 지역 변수. 빠르지만 크기가 제한된다.
- **힙**: 동적 객체. GC/해제 정책에 영향을 받는다.
- **가상 메모리**: 프로세스마다 독립된 주소 공간처럼 보이게 한다.
- **페이지 캐시**: 파일 I/O 성능에 큰 영향을 준다.

### I/O

- 디스크와 네트워크는 CPU보다 느리다.
- blocking I/O는 스레드를 기다리게 한다.
- non-blocking/event-driven I/O는 적은 스레드로 많은 연결을 처리할 수 있지만 코드 복잡도가 올라간다.

## 6. 동시성과 병렬성

| 개념 | 의미 |
|---|---|
| 동시성 | 여러 작업이 겹쳐 진행되는 구조 |
| 병렬성 | 여러 작업이 실제로 동시에 실행되는 것 |
| Race Condition | 실행 순서에 따라 결과가 바뀌는 버그 |
| Deadlock | 서로가 가진 자원을 기다리며 멈춤 |
| Starvation | 특정 작업이 계속 기회를 얻지 못함 |
| Backpressure | 생산 속도를 소비 가능 속도에 맞추는 제어 |

### 안전 원칙

1. 공유 mutable state를 줄인다.
2. 락 범위를 최소화한다.
3. 락 순서를 고정한다.
4. 타임아웃 없이 외부 자원을 기다리지 않는다.
5. 멱등성과 재시도를 함께 설계한다.

## 7. 컴파일러·런타임·언어

- **컴파일**: 사람이 쓴 코드를 기계 또는 VM이 실행 가능한 형태로 바꾼다.
- **인터프리트**: 실행 중 해석한다.
- **JIT**: 실행 중 자주 쓰는 코드를 최적화한다.
- **타입 시스템**: 잘못된 상태를 표현하기 어렵게 만드는 도구다.
- **런타임**: GC, 스케줄러, 표준 라이브러리, 예외 처리 등을 제공한다.

### 좋은 코드의 CS적 기준

- 불변 조건이 타입과 테스트로 보호된다.
- 입력 크기와 실패 모드가 명시되어 있다.
- 예외·타임아웃·취소가 설계되어 있다.
- 관측 가능한 로그와 메트릭이 있다.

## 8. 실무 체크리스트

- [ ] 이 코드의 입력 크기 상한을 알고 있다.
- [ ] 최악의 경우 DB 쿼리 수와 네트워크 호출 수를 설명할 수 있다.
- [ ] 시간·돈·문자열·타임존 처리 규칙이 명시되어 있다.
- [ ] 동시 실행 시 안전한지 테스트했다.
- [ ] 실패·재시도·중복 호출 시 결과가 깨지지 않는다.
- [ ] 성능 병목을 추측이 아니라 측정으로 확인했다.
