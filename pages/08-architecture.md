# 08. 주문/체결 시스템 아키텍처

[07-order-system-tutorial.md](07-order-system-tutorial)에서 만든 작은 코드가, 실제 회사에서는 어떤 거대한 구조 속에서 도는지 봅니다.

## 전체 그림 (한눈에)

```
[고객]
  │ ① 주문 입력 (MTS/HTS 앱)
  ▼
┌─────────────────────────────────────────┐
│  API Gateway / 접속 서버                   │  ← 인증, 트래픽 분산
└─────────────────────────────────────────┘
  │ ② 주문 요청
  ▼
┌─────────────────────────────────────────┐
│  주문 서버 (Order Server) ★코틀린 자주 여기  │
│   - 주문 검증 (예수금, 호가단위, 장시간)      │
│   - 위험 관리 (한도 체크)                    │
└─────────────────────────────────────────┘
  │ ③ 검증 통과한 주문
  ▼
┌─────────────────────────────────────────┐
│  주문 중계 / FEP (거래소 연결 게이트웨이)     │
└─────────────────────────────────────────┘
  │ ④ 거래소 프로토콜로 송신
  ▼
┌─────────────────────────────────────────┐
│  한국거래소 (KRX)                          │
└─────────────────────────────────────────┘
  │ ⑤ 체결/미체결 결과 회신
  ▼
   ↑ 거꾸로 올라오며 ───────────────┐
                                    │
┌──────────────┐   ┌──────────────┐ │
│ 원장 서버      │←──│ 체결 처리      │←┘
│ (잔고/예수금)  │   │ (Execution)   │
└──────────────┘   └──────────────┘
  │ ⑥ 잔고 변경
  ▼
[고객에게 체결 알림 push] + [DB 저장] + [실시간 시세 반영]
```

## 각 구성요소 상세

### ① MTS/HTS (프론트)
고객이 보는 앱/PC 프로그램. 백엔드 개발자는 보통 여기에 **API를 제공**하는 입장.

### ② API Gateway / 접속 서버
수많은 동시 접속을 받아내는 첫 관문. 인증(세션), 부하 분산, 보안. **9시 트래픽 폭증**을 버티는 게 핵심.

### ③ 주문 서버 (Order Server) — ★ 코틀린이 자주 활약
[튜토리얼의 `validateOrder`가 실제로 사는 곳](07-order-system-tutorial).
- 주문 검증 (예수금/증거금, 호가단위, 수량, 장 운영시간)
- **리스크 관리**: 1회 주문 한도, 미수/신용 한도, 이상거래 차단
- 주문 번호 채번, 로그 기록(감사 추적)
- **저지연(low-latency)**이 생명. 0.001초 차이로 체결 가격이 달라짐.

### ④ 주문 중계 / FEP (Front-End Processor)
거래소와 직접 연결되는 게이트웨이. 거래소 전용 프로토콜로 변환해 송신. 세션/순번/재전송 관리.

### ⑤ 거래소 (KRX)
실제 매수/매도를 맞춰주는 매칭 엔진. 체결되면 결과 회신.

### ⑥ 체결 처리 + 원장 서버
[튜토리얼의 `applyExecution`처럼](07-order-system-tutorial) 잔고/예수금 반영 → DB 저장 → 고객 알림.
**원장(Ledger)**은 "고객 자산의 진실의 원천(Source of Truth)". 가장 신중하게 다룸.

## 개발자가 마주하는 어려운 문제들

| 문제 | 왜 어려운가 | 어떻게 다루나 |
|------|------------|--------------|
| **동시성** | 같은 계좌에 매수/매도 동시 진입 시 잔고 꼬임 | DB 트랜잭션, 락, 계좌별 순차 처리 |
| **저지연** | 주문은 ms 단위로 빨라야 함 | 메모리 캐시, 비동기(코루틴), 큐 |
| **정합성** | 주문/체결/원장이 100% 일치해야 함 | 멱등성, 거래 ID 추적, 대사 배치 |
| **장애 복구** | 서버가 죽어도 주문 유실 금지 | 메시지 큐(Kafka), 이중화 |
| **대량 트래픽** | 9시 동시호가 주문 폭증 | 부하 분산, 비동기 처리 |

## 자주 쓰이는 기술 스택 (코틀린 백엔드)

```
언어/프레임워크 : Kotlin + Spring Boot
비동기/동시성   : Kotlin Coroutines, Reactor(WebFlux)
메시지 큐       : Apache Kafka (주문/체결 이벤트 흐름)
캐시           : Redis (시세, 세션, 잔고 캐싱)
DB             : Oracle / PostgreSQL (원장은 보통 관계형 DB)
빌드           : Gradle (build.gradle.kts)
모니터링       : Grafana, Prometheus, 로그 추적
```

> 💡 **Kafka(메시지 큐)** 개념을 미리 가볍게 봐두면 좋습니다. "주문이 들어오면 큐에 쌓고, 여러 서버가 나눠서 처리한다"는 개념이 증권사 이벤트 흐름의 핵심입니다.

## 작은 코드 ↔ 큰 그림 연결

| 튜토리얼 코드 | 실제 시스템 위치 |
|--------------|-----------------|
| `validateOrder()` | ③ 주문 서버의 검증 로직 |
| `applyExecution()` | ⑥ 체결 처리 + 원장 서버 |
| `Account.cash`, `positions` | 원장 DB의 예수금/잔고 |
| `BigDecimal` 계산 | 모든 금액 계산의 표준 |
