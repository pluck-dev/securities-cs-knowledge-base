# 59. 네트워크와 웹 프로토콜

> 프론트엔드 요청, 백엔드 API, 인프라 라우팅, 장애 분석은 모두 네트워크 위에서 일어납니다.

## 1. 요청이 이동하는 길

```
브라우저/앱 → DNS → CDN/WAF → Load Balancer → Reverse Proxy → App → DB/Cache/Queue
```

각 단계마다 지연, 실패, 보안 정책, 로그 위치가 다릅니다. 장애가 나면 "서버가 안 돼요"가 아니라 어느 구간이 실패했는지 좁혀야 합니다.

## 2. TCP/IP 기본

| 계층 | 핵심 | 실무 증상 |
|---|---|---|
| IP | 주소 지정과 라우팅 | 특정 대역만 접속 불가 |
| TCP | 신뢰성 있는 연결 | 연결 지연, 재전송, TIME_WAIT |
| UDP | 연결 없는 전송 | 손실 가능, 낮은 지연 |
| TLS | 암호화와 인증서 | 인증서 만료, SNI, 핸드셰이크 지연 |
| HTTP | 애플리케이션 프로토콜 | 상태코드, 헤더, 캐시, CORS |

### TCP에서 알아야 할 것

- 3-way handshake 후 데이터가 오간다.
- 패킷 손실이 있으면 재전송하고 지연이 튄다.
- 많은 짧은 연결은 handshake 비용과 포트 고갈을 만들 수 있다.
- Keep-Alive와 connection pooling은 서버·클라이언트 모두에서 중요하다.

## 3. DNS

- 도메인을 IP로 바꾼다.
- TTL 때문에 변경이 즉시 전파되지 않을 수 있다.
- `A/AAAA`는 IP, `CNAME`은 다른 이름, `MX`는 메일, `TXT`는 검증·정책에 자주 쓴다.
- 장애 분석 시 `dig`, `nslookup`, 운영체제 DNS 캐시, 브라우저 캐시를 구분한다.

## 4. TLS/HTTPS

TLS는 세 가지를 제공한다.

1. **암호화**: 중간자가 내용을 볼 수 없게 한다.
2. **무결성**: 전송 중 변조를 감지한다.
3. **서버 인증**: 접속한 서버가 인증서의 주체인지 확인한다.

운영 체크리스트:

- 인증서 만료 알림이 있는가?
- 중간 인증서 체인이 올바른가?
- TLS 종료 지점이 LB인지 앱인지 명확한가?
- 내부 구간도 암호화가 필요한가?

## 5. HTTP 핵심

### 메서드

| 메서드 | 의미 | 멱등성 |
|---|---|---|
| GET | 조회 | 있음 |
| POST | 생성/명령 | 보통 없음 |
| PUT | 전체 교체 | 있음 |
| PATCH | 부분 변경 | 설계에 따라 다름 |
| DELETE | 삭제 | 있음으로 설계하는 것이 일반적 |

### 상태 코드

- **2xx**: 성공. `200`, `201`, `204`를 구분한다.
- **3xx**: 리다이렉트와 캐시.
- **4xx**: 클라이언트 요청 문제. 인증 실패, 권한 없음, 검증 실패.
- **5xx**: 서버 또는 의존성 문제. 재시도 정책이 필요하다.

### 헤더

- `Content-Type`: 본문 형식.
- `Accept`: 클라이언트가 원하는 응답 형식.
- `Authorization`: 인증 정보.
- `Cache-Control`, `ETag`: 캐시 정책.
- `X-Request-Id` 또는 `traceparent`: 추적 식별자.

## 6. REST, GraphQL, gRPC, WebSocket, SSE

| 방식 | 적합한 곳 | 주의점 |
|---|---|---|
| REST | 일반 CRUD/API | 리소스 모델링, 버전 관리 |
| GraphQL | 화면별 조합 조회 | N+1, 권한, 캐시 복잡도 |
| gRPC | 내부 서비스 간 고성능 RPC | 브라우저 직접 사용 제한, protobuf 스키마 관리 |
| WebSocket | 양방향 실시간 | 연결 상태, 재연결, scale-out |
| SSE | 서버→클라이언트 스트리밍 | 단방향, 프록시 타임아웃 |

## 7. CORS와 브라우저 보안

- CORS는 서버 리소스를 다른 origin에서 읽을 수 있는지 정하는 브라우저 정책이다.
- Origin은 scheme + host + port다.
- 단순 요청이 아니면 preflight `OPTIONS` 요청이 먼저 간다.
- `Access-Control-Allow-Origin: *`와 credentials 조합은 위험하거나 불가능하다.

## 8. 캐시와 CDN

캐시는 빠르게 만들지만 오래된 데이터를 보여줄 수 있다.

- 정적 자산은 파일명 해시 + 장기 캐시.
- API는 데이터 민감도와 변경 빈도에 따라 `Cache-Control`을 정한다.
- CDN 캐시 무효화는 비용과 지연이 있으므로 URL 버저닝을 선호한다.
- 인증별 응답은 공유 캐시에 저장되면 안 된다.

## 9. 네트워크 장애 분석 루틴

1. 클라이언트 오류 메시지와 시간대를 확보한다.
2. DNS 해석이 되는지 확인한다.
3. TCP 연결이 되는지 확인한다.
4. TLS 인증서와 핸드셰이크를 확인한다.
5. HTTP 상태코드와 응답 헤더를 본다.
6. LB/프록시/앱 로그의 request id를 연결한다.
7. 특정 지역·망·브라우저·버전만 문제인지 분리한다.

## 10. 실무 체크리스트

- [ ] 모든 외부 호출에 timeout이 있다.
- [ ] 재시도는 지수 백오프와 최대 횟수가 있다.
- [ ] POST 재시도에는 idempotency key가 있다.
- [ ] API 응답에 request id가 남는다.
- [ ] CORS 허용 origin이 최소화되어 있다.
- [ ] 인증서 만료 모니터링이 있다.
