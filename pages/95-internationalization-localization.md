# 95. 국제화, 현지화, 시간·숫자 처리

> 글로벌 서비스가 아니어도 시간, 통화, 언어, 단위 처리는 실무 버그의 단골 원인입니다.

## 1. i18n과 l10n

- i18n: 여러 언어/지역을 지원할 수 있게 만드는 구조.
- l10n: 특정 언어/지역에 맞게 번역·형식·문화 요소를 적용.
- 번역은 문자열 치환이 아니라 문맥 전달이다.
- 문자열을 코드에 하드코딩하면 확장 비용이 커진다.

## 2. 시간

시간 처리 원칙:

- 저장은 UTC 기반.
- 표시에는 사용자 timezone 적용.
- 날짜와 시각을 구분.
- 영업일과 달력일을 구분.
- DST를 직접 계산하지 말고 라이브러리 사용.
- 서버와 클라이언트 clock skew를 고려.

## 3. 숫자와 통화

- 통화별 소수 자리 수가 다르다.
- 천 단위 구분자와 소수점 기호가 지역마다 다르다.
- 반올림 규칙이 업무마다 다를 수 있다.
- 금액 계산은 부동소수점이 아니라 정수 최소 단위 또는 decimal 사용.
- 환율 적용 시 기준 시각과 출처가 필요하다.

## 4. 텍스트와 정렬

- Unicode normalization.
- byte length와 character length 차이.
- grapheme cluster.
- locale-aware sorting.
- RTL 언어.
- 폰트 fallback.

한글 초성 검색, 일본어 kana, 중국어 간체/번체도 별도 고려가 필요합니다.

## 5. 번역 운영

- key naming.
- placeholder.
- pluralization.
- gender.
- context note.
- 번역 누락 fallback.
- screenshot review.
- 번역 배포 주기.

## 6. 체크리스트

- [ ] 시간 저장/표시 기준이 문서화되어 있다.
- [ ] 통화와 반올림 규칙이 명확하다.
- [ ] 문자열이 코드에서 분리되어 있다.
- [ ] pluralization과 placeholder를 지원한다.
- [ ] byte 길이 제한과 화면 길이 제한을 구분한다.
- [ ] locale별 정렬/검색을 검토했다.
