# 56. 리스크·컴플라이언스·규제 심화 용어집

증권사 백엔드 개발자가 리스크관리·컴플라이언스·규제 시스템을 구축하고 운영하는 데 필요한 핵심 용어를 망라한 참조 사전이다. 각 용어는 시스템 설계와 규제 대응 관점에서 기술되어 있으므로, API 설계·DB 스키마·배치 프로세스 구현 시 직접 활용할 수 있다.

---

## 사전 위험관리 (Pre-Trade Risk Control)

주문이 거래소로 전송되기 전 실시간으로 수행하는 위험 차단 체계. 미들웨어 또는 OMS(Order Management System) 레이어에서 수행되며, 지연(latency) 최소화가 핵심 품질 지표다.

| 용어 | 영문/약어 | 뜻 | 개발 포인트 |
|---|---|---|---|
| 주문한도 | Order Limit | 계좌·고객·시스템 단위로 설정된 주문 허용 범위의 총칭 | Redis 등 인메모리 저장소에서 원자적(atomic) 비교·차감 처리 필요 |
| 1회주문한도 | Single Order Limit | 1건의 주문에 허용되는 최대 수량·금액 | 주문 수신 즉시 동기 검증; 초과 시 즉각 거부(reject) 응답 반환 |
| 1일주문한도 | Daily Order Limit | 당일 누적 주문금액·수량의 상한 | 영업일 기준 초기화(00:00 KST); 누적값 원자적 증가 후 비교 |
| 종목별한도 | Symbol/Issue Limit | 특정 종목에 대한 보유·주문 허용 상한 | 종목코드(ISIN 또는 단축코드)를 키로 한 한도 테이블 관리 |
| 계좌별한도 | Account Limit | 개별 계좌에 설정된 주문·포지션 허용 범위 | 계좌 속성(신용등급, 투자자 유형)에 따른 다단계 한도 매핑 |
| 포지션한도 | Position Limit | 보유 가능한 순포지션(Long/Short)의 최대치 | 체결 이벤트마다 포지션 원장 업데이트 후 한도 재비교 |
| 손실한도 | Loss Limit / Stop-Loss Limit | 일별·누적 손실이 임계치에 도달하면 추가 주문 차단 | 실시간 P&L 계산 엔진과 연동; 한도 도달 시 자동 거래 차단 플래그 세팅 |
| fat-finger 방지 | Fat-Finger Prevention | 오타·입력 실수로 인한 비정상 주문을 사전 차단 | 직전 체결가 대비 ±N% 초과 주문, 수량 자릿수 이상치 등 다중 필터 적용 |
| 가격이상치체크 | Price Sanity Check | 주문 가격이 현재 시장가 대비 허용 범위 내인지 검증 | 기준가(전일종가·현재호가)를 실시간 참조; 범위 파라미터 운영자 설정 가능하도록 설계 |
| 수량이상치체크 | Quantity Sanity Check | 주문 수량이 비정상적으로 크거나 작은지 검증 | 종목별 최소·최대 거래단위, 발행주식수 대비 비율 기준 적용 |
| 호가범위검증 | Price Band Validation | 거래소 가격제한폭(상·하한가) 내에 주문 가격이 있는지 확인 | 거래소 배포 가격제한 데이터 실시간 수신·캐시 후 비교 |
| 미수발생방지 | Short Sale / Unsettled Buy Prevention | 결제 가능 잔고 이상의 매도 또는 매수를 차단 | 예수금·증거금 원장과 실시간 연동; 결제 D+2 기준 가용금액 계산 |
| 증거금사전검증 | Pre-Trade Margin Check | 파생상품 등에서 주문 전 요구 증거금 충족 여부 확인 | 개시증거금·유지증거금 기준으로 가용 자산 평가; 부족 시 즉각 거부 |
| 신용한도 | Credit Limit | 신용거래 허용 금액의 상한 | 신용공여 잔액, 담보 평가액을 실시간 반영한 가용 신용한도 계산 |
| 위험노출한도 | Risk Exposure Limit | 특정 자산군·섹터·거래상대방에 대한 총 익스포저 상한 | 계층적 한도 구조(그룹→법인→계좌) 지원; 한도 초과 시 경보 및 차단 |
| 주문거부 | Order Reject | 한도 초과·검증 실패 시 주문을 처리하지 않고 반환하는 행위 | 거부 사유 코드(RejectReasonCode) 표준화 후 FIX 메시지 또는 API 응답에 포함 |
| 위험차단 | Risk Kill Switch | 긴급 상황 시 특정 계좌·시스템 전체의 주문 기능을 즉시 중단 | 관리자 API 또는 UI에서 단일 명령으로 활성화; 활성화 이력 감사 로그 기록 필수 |
| 자기매매한도 | Proprietary Trading Limit | 자기 계정 거래에 대한 포지션·손실 한도 | 고객 계좌 한도와 분리된 별도 한도 체계; 준법감시인 승인 절차 연동 |
| 집중도한도 | Concentration Limit | 단일 종목·섹터에 포트폴리오 대비 허용 비중 상한 | 포트폴리오 총 평가액 기준 비율 계산; 일중 지속 모니터링 |

## 시장 리스크·평가 (Market Risk & Valuation)

보유 포지션과 시장 변동에 따른 위험을 측정·평가하는 체계. 일반적으로 배치(batch) 및 실시간 스트리밍 두 가지 경로로 운영된다.

| 용어 | 영문/약어 | 뜻 | 개발 포인트 |
|---|---|---|---|
| 시장위험 | Market Risk | 시장 가격 변동(주가·금리·환율·상품가)으로 인한 손실 위험 | VaR 계산 엔진, 시나리오 분석 모듈과 연계; 종목별 베타·변동성 데이터 필요 |
| 신용위험 | Credit Risk | 거래상대방의 채무불이행으로 인한 손실 위험 | 거래상대방별 신용등급·익스포저 집계; CVA(신용가치조정) 계산 연동 |
| 운영위험 | Operational Risk | 내부 프로세스·시스템·인력 실패 또는 외부 사건으로 인한 손실 | 장애 이벤트 기록, SLA 모니터링, BCP(업무연속성계획) 연동 |
| 유동성위험 | Liquidity Risk | 자산을 적시에 적정 가격으로 매도하지 못할 위험 | 종목별 일평균거래량(ADTV) 대비 보유비율 모니터링; 유동성 스코어 테이블 관리 |
| VaR | Value at Risk | 주어진 신뢰수준(99% 등)·보유기간에서 발생 가능한 최대 손실 추정치 | Historical VaR, Parametric VaR, Monte Carlo 방법론 선택; 일별 배치 계산 후 리포트 저장 |
| 스트레스테스트 | Stress Test | 극단적 시장 시나리오(금융위기 재현 등) 하에서 포트폴리오 손실을 시뮬레이션 | 시나리오 파라미터 DB 관리; 결과 리포트 자동 생성 및 경영진 대시보드 연동 |
| 익스포저 | Exposure | 특정 위험 요인에 대해 노출된 포지션의 시장가치 합계 | 자산 유형별 익스포저 집계 API; 통화별 환산 시 기준환율 데이터 연동 |
| 포지션평가 | Position Valuation / Mark-to-Market | 보유 포지션을 현재 시장가격으로 평가하는 행위 | 시가평가(MTM)와 원가평가 병행 관리; 종가 확정 후 배치 재계산 |
| 일일손익 (P&L) | Daily Profit & Loss | 하루 동안의 실현·미실현 손익 합계 | 체결 기반 실현손익 + 포지션 MTM 변동액; T+0 실시간 누적 및 T 배치 확정 |
| 민감도 | Greeks / Sensitivity | 기초자산 가격·변동성·금리 변화에 대한 파생상품 가치 변화율(델타·감마·베가·세타 등) | 파생상품 프라이싱 엔진 연동; Greeks 값 DB 저장 후 리스크 집계 |
| 한도소진율 | Limit Utilization Rate | 설정된 한도 대비 현재 사용량의 비율(%) | (현재값/한도) × 100; 임계치(80%, 90%, 100%) 도달 시 단계별 경보 발송 |
| 위험가중자산 | Risk-Weighted Assets (RWA) | 자산별 신용위험 가중치를 적용한 자산 합계; 자본적정성 계산 기준 | 바젤 기준 위험가중치 테이블 관리; 월별 규제보고 산출 배치 |
| 영업용순자본비율 | Net Capital Ratio (NCR) | 영업용순자본을 총위험액으로 나눈 비율; 증권사 건전성 지표 | 금감원 규정상 100% 이상 유지; 일별 산출 및 보고 자동화 |
| 자기자본규제 | Capital Adequacy Regulation | 금융당국이 요구하는 최소 자기자본 보유 규제 | 규제 변경 시 계산 로직 즉시 반영 가능한 파라미터 기반 설계 권장 |
| 백테스팅 | Backtesting | VaR 모델의 정확성을 과거 실제 손익과 비교·검증하는 과정 | 예외(Exception) 발생 횟수 집계; 신호등(traffic light) 기준 모델 적합성 판정 로직 구현 |
| CVA | Credit Valuation Adjustment | 거래상대방 신용위험을 반영한 파생상품 가치 조정 | 거래상대방 PD(부도확률)·LGD(손실비율) 파라미터 관리; 포트폴리오 단위 집계 |
| 순포지션 | Net Position | 동일 종목의 매수·매도 포지션을 상계한 순 보유량 | 체결 원장에서 실시간 집계; 롱(+)/숏(-) 부호 처리 명확화 |
| 마진콜 | Margin Call | 증거금 부족 시 추가 증거금 납입을 요구하는 절차 | 유지증거금 이하 하락 감지 시 자동 통지(SMS·이메일); 미납 시 강제청산 프로세스 연동 |

## 이상거래·불공정거래 탐지 (Abnormal & Unfair Trade Detection)

시장 질서를 해치는 거래 패턴을 실시간 또는 사후에 탐지하고 보고하는 체계. 거래소·금감원 협력 보고 의무가 수반된다.

| 용어 | 영문/약어 | 뜻 | 개발 포인트 |
|---|---|---|---|
| 이상거래탐지 | Fraud Detection System (FDS) | 비정상적 거래 패턴을 탐지하는 시스템 전반 | 룰 기반(Rule-Based)과 ML 기반 모델 병행; 실시간 스트리밍 처리(Kafka 등) 권장 |
| 시세조종 | Market Manipulation | 인위적인 매매를 통해 시세를 의도적으로 변동시키는 행위 | 특정 계좌의 시세 영향도(가격충격) 분석; 단기 급등락 패턴 룰 정의 |
| 통정매매 | Prearranged Trading | 사전 협의된 상대방과 매수·매도를 맞춰 거래하는 불공정 행위 | 동일 IP, 동일 시간대, 상호 대응 주문 패턴 탐지 로직 구현 |
| 자전거래 | Cross Trading | 동일 주체가 매수·매도를 동시에 내어 체결시키는 거래 | 동일 계좌군 또는 연계 계좌 간 상호 체결 감지; 계좌 연관 관계 그래프 DB 활용 가능 |
| 가장매매 | Wash Trading | 실질적 소유권 이전 없이 매매한 것처럼 꾸미는 행위 | 자전거래 탐지와 유사; 체결 후 원상복구 패턴 추가 감지 |
| 허수주문 | Layering / Spoofing | 체결 의사 없이 대량 주문을 제출 후 즉시 취소해 시세를 오도하는 행위 | 주문 제출→취소 시간 간격, 취소율(Cancel Rate) 임계치 룰 적용 |
| 미공개정보이용 | Insider Trading | 공시 전 중요 내부 정보를 이용한 거래 | 공시 이전 비정상 거래량 급증 탐지; 공시 DB와 거래 시각 조인 분석 |
| 부정거래 | Fraudulent Trading | 사기·기망 등 불법 수단을 이용한 거래 행위 | 광범위한 이상 패턴 탐지; 법무팀 보고 워크플로 자동화 |
| 스캘핑 | Scalping | 단기 시세 차익을 목적으로 매우 짧은 시간 내 반복 매매하는 행위(합법적 전략이나 규제 맥락에서 감시 대상) | 초단위 거래 빈도·회전율 분석; 계좌별 평균 보유 시간 집계 |
| 선행매매 (프론트러닝) | Front-Running | 고객 주문 정보를 미리 알고 자기 계정으로 먼저 매매하는 행위 | 직원 계좌와 고객 주문 타임스탬프 비교; 정보교류차단 위반 여부 연계 점검 |
| 시장교란 | Market Disruption | 시장의 정상적인 가격 형성을 방해하는 행위 총칭 | 복수 탐지 룰 조합 결과로 판정; 거래소 이상거래 보고 포맷에 맞춰 자동 생성 |
| 단주매매패턴 | Odd-Lot Trading Pattern | 1주 단위 소량 주문을 반복해 시세를 탐색하거나 조작하는 패턴 | 단주 주문 비율·빈도 임계치 설정; 정상 소액 투자자와 패턴 구분 알고리즘 필요 |
| 알람/경보 | Alert / Warning | 탐지 룰 발동 시 담당자에게 전달되는 통지 | 경보 우선순위(Critical/High/Medium/Low) 분류; 알람 피로 방지를 위한 중복 제거 로직 |
| 혐의거래 | Suspicious Transaction | 불공정거래로 의심되어 추가 조사가 필요한 거래 | 혐의거래 관리 DB 설계(상태: 탐지→조사중→종결); 규제기관 보고 연동 |
| 이상거래보고 | Abnormal Trading Report | 거래소·금감원에 제출하는 이상거래 탐지 결과 보고서 | 거래소 규정 포맷(XML/CSV) 자동 생성; 제출 이력 및 회신 관리 |
| 계좌연관분석 | Account Linkage Analysis | 동일인이 운용하는 것으로 의심되는 계좌들을 연결해 분석 | 공통 IP, 단말기 ID, 입금 경로 기반 그래프 클러스터링 |
| 거래회전율 | Turnover Rate | 계좌 내 자산 대비 매매 거래량 비율; 과당매매 판단 지표 | 일·월 단위 집계; 증권사 자기매매 대비 고객 계좌 별도 분리 계산 |

## AML·KYC·고객보호 (AML, KYC & Customer Protection)

자금세탁 방지 및 고객 신원 확인 체계. 금융정보분석원(FIU)·FATF 기준을 준수해야 하며, 온보딩 시스템과 거래 모니터링 시스템 양쪽에 걸쳐 있다.

| 용어 | 영문/약어 | 뜻 | 개발 포인트 |
|---|---|---|---|
| 자금세탁방지 | Anti-Money Laundering (AML) | 범죄 수익을 합법적 자금처럼 위장하는 행위를 탐지·방지하는 제도 | 거래 모니터링 시스템(TMS) 구축; 룰·ML 하이브리드 탐지 엔진 설계 |
| 고객확인 | Know Your Customer (KYC) | 금융서비스 제공 전 고객 신원·거래 목적을 확인하는 절차 | 온보딩 API에서 신분증 진위 확인(OCR+진위 API) 및 결과 저장 |
| 고객확인의무 | Customer Due Diligence (CDD) | KYC의 법적 의무 형태; 신원 확인·실제소유자 확인·거래 목적 파악 포함 | 계좌 개설 시 CDD 완료 여부 플래그 관리; 미완료 시 특정 서비스 잠금 |
| 강화된확인 | Enhanced Due Diligence (EDD) | 고위험 고객(PEP, 고위험 국가 거주자 등)에 대해 심화 검증을 수행하는 절차 | CDD 결과 및 위험 점수 기반 자동 EDD 트리거; 담당자 승인 워크플로 연동 |
| 의심거래보고 | Suspicious Transaction Report (STR) | 자금세탁·테러자금 의심 거래를 FIU에 보고하는 문서 | 보고 기한(영업일 3일 내); FIU 전자보고 시스템(goAML) API 연동 |
| 고액현금거래보고 | Currency Transaction Report (CTR) | 1일 1천만 원 이상 현금 거래를 FIU에 자동 보고하는 제도 | 동일인 분산 입금(구조화) 탐지; 자동 집계 및 기한 내 전송 배치 |
| 실제소유자 | Ultimate Beneficial Owner (UBO) | 법인·신탁 등의 배후에서 실질적으로 지배하는 자연인 | 25% 이상 지분 기준 UBO 식별; 법인 고객 온보딩 시 지배구조 수집 API 설계 |
| 요주의인물 | Watch List | 자금세탁·테러·제재 관련 주의 대상 인물 목록 | OFAC·UN·EU·국내 제재 리스트 주기적 업데이트; 고객 정보와 배치·실시간 매칭 |
| 제재대상 | Sanctioned Entity | 국제·국내 제재를 받아 거래가 금지된 개인 또는 단체 | 제재 리스트 명중 시 즉각 거래 차단 및 담당자 통보; 오탐(false positive) 검토 프로세스 필요 |
| 적합성원칙 | Suitability Rule | 금융투자상품이 고객의 투자 목적·재무상황·위험 성향에 적합한지 확인할 의무 | 투자자 성향 설문 결과와 상품 위험등급 매핑; 부적합 경고 로직 및 고객 서명 저장 |
| 적정성원칙 | Appropriateness Rule | 비권유(자기 선택) 거래에서도 고객에게 위험을 고지하고 적정성을 확인하는 의무 | 적합성과 달리 권유 없이도 적용; 고객 확인 화면 UX 및 동의 기록 저장 |
| 설명의무 | Duty to Explain | 금융상품 판매 시 핵심 정보를 고객이 이해할 수 있도록 설명해야 하는 법적 의무 | 설명확인서 전자 서명 및 원본 보관(5년); 설명 항목 체크리스트 API 관리 |
| 투자자성향분석 | Investor Risk Profiling | 고객의 위험 감수 성향을 분류(안정형~공격투자형)하는 절차 | 설문 응답 점수화 알고리즘; 유효기간(통상 1년) 만료 시 재설문 요청 로직 |
| 전문투자자 | Professional Investor | 금융투자에 관한 전문 지식·경험을 갖춘 투자자(기관, 지정 개인 등) | 전문투자자 지정 요건 DB 관리; 일반투자자 대비 규제 완화 상품 접근 권한 분리 |
| 일반투자자 | Retail Investor | 전문투자자가 아닌 일반 개인 투자자 | 판매 가능 상품 제한, 적합성·적정성 의무 전면 적용 |
| 부적합상품 | Unsuitable Product | 투자자 성향 분석 결과 해당 고객에게 맞지 않는 상품 | 부적합 상품 주문 시 경고 팝업 및 재확인 절차; 고객 동의 기록 보존 |
| AML 위험평가 | AML Risk Assessment | 고객·상품·채널·지역별 자금세탁 위험을 점수화하는 절차 | 위험 점수 모델 설계(고객 속성·거래 패턴·지역 가중치); 주기적 재산출 배치 |
| 테러자금조달방지 | Counter Financing of Terrorism (CFT) | 테러 행위에 자금이 공급되는 것을 차단하는 제도 | STR 보고와 동일 채널 활용; 테러 관련 제재 리스트 별도 관리 권장 |

## 규제·내부통제·감사 (Regulation, Internal Control & Audit)

금융 규제 준수와 내부 통제 체계를 뒷받침하는 시스템 요소. 감사 추적과 기록 보존은 모든 시스템에 공통으로 적용되어야 한다.

| 용어 | 영문/약어 | 뜻 | 개발 포인트 |
|---|---|---|---|
| 자본시장법 | Financial Investment Services and Capital Markets Act (FSCMA) | 증권·파생·집합투자 등 자본시장 전반을 규율하는 국내 기본법 | 법 개정 반영을 위한 규제 파라미터 관리 체계 필수; 법률 변경 이력 추적 |
| 전자금융거래법 | Electronic Financial Transactions Act (EFTA) | 전자적 수단을 통한 금융거래의 안전성·신뢰성 확보를 위한 법 | 전자금융사고 보고 의무(24시간 내); 접근매체 분실·도용 처리 프로세스 구현 |
| 개인정보보호법 | Personal Information Protection Act (PIPA) | 개인정보의 수집·이용·보호에 관한 기본법 | 개인정보 처리 방침 관리; 보유기간 만료 데이터 자동 삭제 스케줄러 구현 |
| 신용정보법 | Credit Information Use and Protection Act | 신용정보의 수집·이용·유통·보호에 관한 법 | 신용정보 제공·활용 동의 체계; 마이데이터 API 연동 시 의무 준수 사항 확인 |
| 금융소비자보호법 | Financial Consumer Protection Act | 금융소비자의 권익 보호를 위한 판매 규제(6대 판매 원칙) | 청약철회권·위법계약해지권 처리 프로세스 구현; 기록 보존 의무 |
| 내부통제기준 | Internal Control Standards | 법령 준수·손실 방지를 위해 회사가 내부적으로 마련한 통제 규정 | 내부통제 항목별 점검 결과 기록; 준법감시인 검토 워크플로 시스템화 |
| 준법감시인 | Compliance Officer | 내부통제기준 준수 여부를 감시하는 법적 의무 임직원 | 준법감시인 승인이 필요한 이벤트(한도 변경 등) 전자결재 API 설계 |
| 정보교류차단 (차이니즈월) | Chinese Wall / Information Barrier | 이해충돌을 방지하기 위해 부서 간 정보 공유를 차단하는 제도적 장치 | 시스템 접근 권한 매트릭스로 구현; 차단 벽 위반 시도 로그 자동 기록 |
| 감사추적 | Audit Trail | 시스템 내 모든 중요 행위(주문·체결·설정변경 등)를 시간순으로 기록한 로그 | 불변(immutable) 로그 저장소 사용(WORM 스토리지 또는 블록체인 해시); 조작 방지 설계 |
| 매매기록보존 | Trade Record Retention | 거래 관련 기록을 규정 기간(통상 10년) 동안 보존하는 의무 | 콜드 스토리지 아카이빙 파이프라인; 기간별 데이터 계층화(hot→warm→cold) |
| 녹취 | Call Recording | 투자 권유 관련 통화를 녹음·보존하는 의무 | 녹취 파일 메타데이터(고객ID·시각·담당자) 연동; 10년 보존 및 검색 API |
| 전자금융감독규정 | Electronic Financial Supervision Regulation | 금감원이 전자금융업자에게 적용하는 보안·운영 기준 | 정보보호 관리체계(ISMS) 인증 요구사항 시스템 반영; 보안 취약점 정기 점검 |
| 망분리 | Network Separation | 인터넷망과 내부 업무망을 물리적·논리적으로 분리하는 보안 요건 | 망분리 환경에서의 배포·운영 파이프라인 재설계; 데이터 이동 시 승인된 단방향 전송 도구 사용 |
| 접근통제 | Access Control | 시스템·데이터에 대한 접근 권한을 역할·필요 기준으로 제한하는 보안 원칙 | RBAC(역할 기반 접근제어) 또는 ABAC 구현; 권한 부여·회수 이력 감사 로그 |
| 권한관리 | Privilege Management | 임직원·시스템 계정의 접근 권한을 생성·변경·회수하는 체계 | 최소 권한 원칙(Principle of Least Privilege) 적용; 분기별 권한 검토 자동화 |
| 보고의무 | Reporting Obligation | 금감원·거래소·FIU 등 규제기관에 정해진 항목과 기한 내 데이터를 보고하는 의무 | 규제 보고 스케줄러; 보고 실패 시 알람 및 재시도 로직; 제출 이력 DB 관리 |
| 전산 장애 보고 | IT Incident Report | 전자금융 사고 발생 시 금감원에 보고하는 의무 절차 | 장애 감지→분류→보고 자동화 파이프라인; 24시간 이내 보고 기한 준수 타이머 |
| 업무연속성계획 | Business Continuity Plan (BCP) | 재해·장애 상황에서도 핵심 업무를 지속하기 위한 계획 | DR(재해복구) 시스템 자동 절체(Failover); RTO·RPO 목표치 정의 및 정기 훈련 기록 |
| 보안사고 대응 | Security Incident Response | 침해 사고 발생 시 탐지·분석·차단·복구하는 절차 | SIEM 연동; 침해지표(IoC) 자동 업데이트; 대응 플레이북 시스템화 |
| 개인정보 영향평가 | Privacy Impact Assessment (PIA) | 개인정보 처리 시스템의 프라이버시 위험을 사전에 평가하는 절차 | 신규 시스템 구축 시 PIA 체크리스트 통과 요건; 평가 결과 문서 저장 |
| 규제 샌드박스 | Regulatory Sandbox | 혁신 금융서비스를 일정 기간 규제 특례를 받아 운영할 수 있는 제도 | 샌드박스 기간 중 별도 모니터링 지표 수집; 기간 만료 후 정규 규제 전환 절차 구현 |
