# 65. DevOps와 SRE 운영 지식

> 운영은 배포 후에 시작되는 일이 아니라 설계 단계부터 들어가야 하는 품질 속성입니다.

## 1. DevOps와 SRE

- DevOps는 개발과 운영의 경계를 줄여 빠르고 안전하게 변경하는 문화와 실천입니다.
- SRE는 신뢰성을 엔지니어링 문제로 다룹니다.
- 핵심은 "많이 배포하면서도 사용자가 느끼는 장애를 줄이는 것"입니다.

## 2. SLI, SLO, SLA

| 용어 | 의미 |
|---|---|
| SLI | 측정 지표. 예: 성공률, 지연 시간 |
| SLO | 내부 목표. 예: 99.9% 요청이 300ms 이하 |
| SLA | 고객과의 계약. 위반 시 보상 가능 |
| Error Budget | 허용 가능한 실패량 |

SLO가 없으면 모든 장애가 같은 우선순위가 되어 팀이 지칩니다.

## 3. 관측성 3대 축

### Logs

개별 이벤트를 설명합니다. 구조화 로그와 trace id가 중요합니다.

### Metrics

시계열 숫자입니다. 알림과 대시보드에 적합합니다.

### Traces

분산 요청의 경로와 각 구간 지연을 보여줍니다.

## 4. 알림 설계

좋은 알림:

- 사용자가 영향을 받거나 곧 받을 때 울린다.
- 담당자가 바로 행동할 수 있는 정보를 준다.
- 중복·소음이 적다.

나쁜 알림:

- CPU 80% 같은 원인 지표만 보고 무조건 울린다.
- 조치 방법이 없다.
- 밤마다 반복되어 무시된다.

## 5. 배포 전략

| 전략 | 설명 |
|---|---|
| Rolling | 순차적으로 새 버전 교체 |
| Blue/Green | 두 환경 중 트래픽 전환 |
| Canary | 일부 트래픽만 새 버전 |
| Feature Flag | 코드 배포와 기능 공개 분리 |

무중단 배포에는 하위 호환 API, DB 마이그레이션 순서, graceful shutdown이 필요합니다.

## 6. 장애 대응

### Incident lifecycle

1. 탐지.
2. 영향 범위 파악.
3. 완화 조치.
4. 원인 분석.
5. 재발 방지.
6. 사후 리뷰.

### 장애 중 원칙

- 복구가 분석보다 우선이다.
- 지휘자, 기록자, 실행자를 나눈다.
- 추측은 로그와 지표로 검증한다.
- 고객 커뮤니케이션은 사실과 시간 기준으로 한다.

## 7. 포스트모템

좋은 포스트모템은 비난하지 않고 시스템 개선으로 이어집니다.

포함할 내용:

- 타임라인.
- 사용자 영향.
- 탐지 방법.
- 직접 원인과 기여 요인.
- 잘 된 점.
- 개선 액션과 담당자, 기한.

## 8. 용량 계획

- 현재 QPS, 피크 QPS, 성장률을 본다.
- p95/p99 지연과 에러율을 함께 본다.
- DB 커넥션, 큐 적체, 스레드 풀, CPU steal, GC를 확인한다.
- 부하 테스트는 실제 트래픽 패턴과 데이터 분포를 반영해야 한다.

## 9. 운영 자동화

자동화할 것:

- 반복 배포.
- 롤백.
- 인증서 갱신.
- 백업 검증.
- 로그/메트릭 대시보드 생성.
- 런북 기반 진단.

자동화하지 말아야 할 것:

- 이해하지 못한 복구 명령.
- 안전장치 없는 대량 삭제.
- 권한 상승이 필요한 민감 작업.

## 10. SRE 체크리스트

- [ ] 핵심 사용자 흐름별 SLO가 있다.
- [ ] 알림은 사용자 영향 중심이다.
- [ ] 배포는 자동화되어 있고 롤백 가능하다.
- [ ] 장애 대응 런북이 최신이다.
- [ ] 포스트모템 액션이 추적된다.
- [ ] 백업 복구와 DR 훈련을 해봤다.
