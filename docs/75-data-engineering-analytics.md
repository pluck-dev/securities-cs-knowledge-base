# 75. 데이터 엔지니어링과 분석 기반

> 제품 개발자는 운영 DB뿐 아니라 로그, 이벤트, 지표, 분석 파이프라인의 기본 구조도 이해해야 합니다.

## 1. OLTP와 OLAP

| 구분 | OLTP | OLAP |
|---|---|---|
| 목적 | 서비스 트랜잭션 | 분석과 리포팅 |
| 쿼리 | 짧고 빈번 | 크고 복잡 |
| 데이터 | 최신 정합성 중요 | 대량 이력 중요 |
| 예 | 주문 DB, 계좌 DB | 데이터 웨어하우스 |

서비스 DB에 무거운 분석 쿼리를 직접 날리면 운영 장애가 날 수 있습니다.

## 2. 이벤트 로그

좋은 이벤트는 분석과 디버깅 모두에 도움이 됩니다.

포함할 것:

- event name.
- user/account id.
- timestamp.
- source.
- trace/session id.
- 중요한 속성.
- schema version.

주의: 개인정보와 민감정보를 이벤트에 그대로 넣지 않습니다.

## 3. ETL과 ELT

- ETL: 추출, 변환, 적재.
- ELT: 추출, 적재, 웨어하우스 안에서 변환.

현대 데이터 스택은 ELT가 많지만, 민감정보 마스킹이나 품질 검증은 적재 전 필요할 수 있습니다.

## 4. 배치와 스트리밍

| 방식 | 장점 | 단점 |
|---|---|---|
| Batch | 단순, 비용 효율 | 실시간성 낮음 |
| Streaming | 실시간 처리 | 운영 복잡도 |
| Micro-batch | 중간 절충 | 지연과 복잡도 모두 존재 |

## 5. 데이터 품질

- completeness: 누락이 없는가?
- uniqueness: 중복이 없는가?
- validity: 형식이 맞는가?
- consistency: 시스템 간 값이 맞는가?
- freshness: 최신인가?
- accuracy: 실제와 맞는가?

## 6. 스키마 진화

데이터 파이프라인은 스키마 변경에 약합니다.

- 필드 추가는 대체로 안전.
- 필드 삭제와 타입 변경은 위험.
- enum 값 추가도 소비자에 따라 문제.
- schema registry를 사용하면 호환성을 검사할 수 있다.

## 7. 지표 설계

지표는 정의가 생명입니다.

예:

- 활성 사용자: 어떤 기간? 로그인 기준? 이벤트 기준?
- 주문 전환율: 분모와 분자는 무엇인가?
- 장애율: 전체 요청 중 5xx? 비즈니스 실패 포함?

정의가 없으면 팀마다 다른 숫자를 보게 됩니다.

## 8. 개인정보와 데이터 거버넌스

- 데이터 분류: 공개/내부/민감/개인정보.
- 접근 권한 최소화.
- 보관 기간과 파기 정책.
- lineage: 데이터가 어디서 와서 어디로 가는가.
- masking/tokenization.

## 9. 데이터 파이프라인 장애

자주 나는 문제:

- 중복 적재.
- 지연 적재.
- 일부 partition 누락.
- 스키마 변경으로 consumer 실패.
- timezone 기준 불일치.
- 재처리 시 멱등성 없음.

## 10. 체크리스트

- [ ] 운영 DB와 분석 쿼리 부하가 분리되어 있다.
- [ ] 이벤트 스키마와 버전이 정의되어 있다.
- [ ] 데이터 품질 검사가 있다.
- [ ] 지표 정의가 문서화되어 있다.
- [ ] 개인정보가 로그/이벤트에 노출되지 않는다.
- [ ] 재처리와 중복 적재 대응이 있다.
