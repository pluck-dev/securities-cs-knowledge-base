# 11. 심화: 동시성과 잔고 정합성

"같은 계좌에 동시에 주문이 들어오면 잔고가 꼬인다" — 증권사에서 가장 무서운 버그입니다.
실행 예제: [`examples/RaceCondition.kt`](../examples/RaceCondition.kt) (검증 완료)

## 왜 잔고가 꼬이나? (경쟁 상태, Race Condition)

`balance -= 1` 한 줄은 사실 **3단계**로 실행됩니다.

```
1. 읽기:  현재 balance를 읽는다       (예: 100)
2. 계산:  100 - 1 = 99
3. 쓰기:  balance에 99를 저장한다
```

두 스레드 A, B가 동시에 실행되면:

```
A: 읽기 100 ─┐
B: 읽기 100 ─┤  둘 다 100을 읽음 (아직 아무도 안 씀)
A: 99 저장   │
B: 99 저장 ──┘  ← B도 100 기준으로 계산해서 99 저장 → 1번의 차감이 사라짐!
```

이걸 **갱신 유실(lost update)**이라 합니다.

## 실제 결과 (예제 출력)

10만 번씩 두 스레드가 동시에 차감:

```
[A. 잠금 없음]
  기대 잔고: 0
  실제 잔고: 57474   ← 차감이 무더기로 유실됨!

[B. 잠금 있음 (synchronized)]
  기대 잔고: 0
  실제 잔고: 0       ← 항상 정확
```

> 숫자(57474)는 실행할 때마다 다릅니다. "예측 불가능하게 틀린다"는 게 경쟁 상태의 무서운 점입니다.

## 해법 1 — 잠금(Lock)으로 직렬화

```kotlin
class SafeAccount(var balance: Long) {
    private val lock = Any()
    fun withdraw(amount: Long) {
        synchronized(lock) {   // 한 번에 한 스레드만 진입 → 읽기·계산·쓰기가 안 쪼개짐
            balance -= amount
        }
    }
}
```

## 실무에서 쓰는 방법들

| 방법 | 설명 | 언제 |
|------|------|------|
| **DB 트랜잭션** | 여러 작업을 "전부 성공 또는 전부 취소"로 묶음 | 기본 |
| **비관적 락 (Pessimistic Lock)** | DB 행을 잠그고 작업 (`SELECT ... FOR UPDATE`) | 충돌 잦을 때 |
| **낙관적 락 (Optimistic Lock)** | 버전 번호로 충돌 감지, 충돌 시 재시도 | 충돌 드물 때 |
| **계좌별 순차 처리** | 같은 계좌 주문을 하나의 큐/스레드로 직렬화 | 고성능 트레이딩 |
| **단일 스레드 모델** | 핵심 매칭을 한 스레드로 (예: LMAX Disruptor) | 초저지연 거래소 |

## 트랜잭션의 ACID (DB 동시성의 기본)

| 글자 | 뜻 | 증권 예시 |
|------|-----|----------|
| **A**tomicity (원자성) | 전부 되거나 전부 안 됨 | 출금 + 매수반영이 함께 성공/실패 |
| **C**onsistency (일관성) | 규칙을 항상 만족 | 잔고는 음수가 될 수 없다 |
| **I**solation (격리성) | 동시 거래가 서로 간섭 안 함 | 위 race condition 방지 |
| **D**urability (지속성) | 성공하면 영구 보존 | 체결 기록은 사라지지 않음 |

## 멱등성(Idempotency) — 중복 방지

네트워크 오류로 같은 주문이 두 번 전송될 수 있습니다. **두 번 처리되면 안 됩니다.**

```
- 주문에 고유 ID(예: clientOrderId)를 부여
- 서버는 "이미 처리한 ID인지" 확인 후, 중복이면 무시하고 기존 결과 반환
```

## 체크리스트

- [ ] 공유 상태(잔고, 잔량)를 여러 스레드가 동시에 바꾸지 않나?
- [ ] 바꾼다면 락/트랜잭션으로 보호했나?
- [ ] 중복 요청에 멱등성 키가 있나?
- [ ] 트랜잭션 범위가 적절한가? (너무 넓으면 성능 저하, 너무 좁으면 정합성 깨짐)
