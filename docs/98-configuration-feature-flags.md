# 98. 설정 관리와 Feature Flag

> 설정과 플래그는 배포 없이 동작을 바꾸는 강력한 도구지만, 통제하지 않으면 장애 원인이 됩니다.

## 1. 설정의 종류

- build-time config.
- deploy-time config.
- runtime config.
- secret.
- experiment flag.
- ops kill switch.
- permission flag.

각 설정은 소유자, 기본값, 변경 절차, 감사 로그가 필요합니다.

## 2. 12-Factor Config

설정은 코드와 분리하고 환경에서 주입합니다.

주의:

- secret을 repo에 넣지 않는다.
- 환경별 차이를 코드 분기로 만들지 않는다.
- 기본값이 안전해야 한다.
- 설정 변경이 언제 반영되는지 명확해야 한다.

## 3. Feature Flag 유형

| 유형 | 용도 |
|---|---|
| Release flag | 기능 공개 제어 |
| Experiment flag | A/B 테스트 |
| Ops flag | 장애 시 기능 끄기 |
| Permission flag | 특정 사용자/조직 허용 |
| Migration flag | 점진적 전환 |

각 플래그는 만료일이 있어야 합니다.

## 4. 플래그 평가 위치

- 클라이언트 평가: UI 제어에 편하지만 조작 가능.
- 서버 평가: 보안과 일관성에 유리.
- edge 평가: latency 개선 가능.

권한과 보안에 관련된 결정은 서버에서 최종 판단해야 합니다.

## 5. 설정 장애 패턴

- 잘못된 기본값.
- 환경 변수 이름 오타.
- 플래그 삭제 누락.
- 캐시된 설정과 최신 설정 불일치.
- 일부 인스턴스만 변경.
- 설정 변경 감사 로그 없음.

## 6. 체크리스트

- [ ] 설정 소유자와 설명이 있다.
- [ ] secret과 일반 설정이 분리되어 있다.
- [ ] 플래그 만료일과 제거 계획이 있다.
- [ ] 설정 변경 감사 로그가 있다.
- [ ] 잘못된 설정 시 안전한 기본값으로 실패한다.
- [ ] 권한 관련 플래그는 서버에서 검증한다.
