# 104. 클린 코드와 유지보수성 심화

> 좋은 코드는 짧은 코드가 아니라 변경 이유가 명확하고 실패를 안전하게 다루는 코드입니다.

## 1. 이름

이름은 가장 중요한 문서입니다.

- 도메인 용어 사용.
- 축약어 남용 금지.
- boolean은 긍정문 선호.
- side effect가 있으면 이름에 드러낸다.
- 단위와 통화를 이름이나 타입에 포함.

## 2. 함수와 모듈

- 한 함수는 한 추상화 수준을 유지한다.
- 입력과 출력이 명확해야 한다.
- 숨은 global state를 줄인다.
- side effect를 격리한다.
- 모듈은 변경 이유가 같은 것끼리 묶는다.

## 3. 오류 처리

- 예외를 삼키지 않는다.
- 도메인 오류와 시스템 오류를 구분한다.
- 복구 가능한 오류와 불가능한 오류를 구분한다.
- 오류 메시지에는 조치 가능 정보가 있어야 한다.
- 민감정보를 오류에 포함하지 않는다.

## 4. 중복과 추상화

중복 제거는 항상 좋은 것이 아닙니다.

- 우연한 중복은 놔둘 수 있다.
- 같은 이유로 바뀌는 중복은 추상화한다.
- 추상화는 이름과 경계가 명확해야 한다.
- 너무 이른 공통화는 변경을 어렵게 한다.

## 5. 리팩터링 루틴

1. 테스트로 현재 동작 고정.
2. 작은 변경.
3. 즉시 테스트.
4. 이름 개선.
5. 중복 제거.
6. 죽은 코드 삭제.
7. 성능과 로그 확인.

## 6. 체크리스트

- [ ] 이름이 도메인 언어와 맞다.
- [ ] side effect가 격리되어 있다.
- [ ] 오류가 명시적으로 처리된다.
- [ ] 중복 제거 이유가 명확하다.
- [ ] 리팩터링 전후 테스트가 있다.
- [ ] 삭제 가능한 코드가 남아 있지 않다.
