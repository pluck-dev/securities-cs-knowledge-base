# 60. 백엔드 엔지니어링 필수 지식

> 백엔드는 HTTP 요청을 받아 비즈니스 규칙, 데이터 일관성, 보안, 운영 가능성을 책임지는 영역입니다.

## 1. 백엔드의 기본 책임

1. 입력을 검증한다.
2. 인증된 사용자가 권한 있는 작업만 하게 한다.
3. 도메인 규칙을 일관되게 적용한다.
4. 데이터 저장과 트랜잭션을 안전하게 처리한다.
5. 외부 시스템 실패에 견딘다.
6. 로그·메트릭·트레이스로 운영자가 이해할 수 있게 만든다.

## 2. 계층 구조

```
Controller/API → Application Service → Domain Model → Repository/Gateway → DB/External API
```

- Controller는 HTTP 변환과 얇은 검증에 집중한다.
- Application Service는 유스케이스 흐름을 조정한다.
- Domain Model은 핵심 규칙과 불변 조건을 지킨다.
- Repository는 저장소 세부 구현을 숨긴다.
- Gateway/Client는 외부 API와의 통신을 캡슐화한다.

## 3. API 설계

### 좋은 API의 조건

- 리소스와 행위가 명확하다.
- 요청·응답 스키마가 문서화되어 있다.
- 검증 오류가 필드 단위로 설명된다.
- 페이지네이션·정렬·필터 기준이 일관된다.
- breaking change 정책과 버전 전략이 있다.

### 오류 응답 예시

```json
{
  "code": "ORDER_PRICE_INVALID",
  "message": "주문 가격이 허용 범위를 벗어났습니다.",
  "traceId": "01J...",
  "details": [
    { "field": "price", "reason": "must be positive tick price" }
  ]
}
```

## 4. 인증과 인가

- **Authentication**: 누구인지 확인한다.
- **Authorization**: 무엇을 할 수 있는지 판단한다.
- 세션, JWT, OAuth2/OIDC의 차이를 이해한다.
- 토큰은 저장 위치와 만료, 재발급, 폐기 전략이 중요하다.
- 권한 검사는 프론트엔드가 아니라 서버에서 최종 수행한다.

## 5. 트랜잭션과 일관성

트랜잭션은 비즈니스 불변 조건의 경계입니다.

- 한 주문 생성에서 잔고 예약과 주문 저장이 함께 성공하거나 함께 실패해야 할 수 있다.
- 외부 API 호출은 DB 트랜잭션 안에 오래 넣지 않는 것이 좋다.
- 격리 수준에 따라 dirty read, non-repeatable read, phantom read가 달라진다.
- 낙관적 락은 충돌이 드문 경우, 비관적 락은 충돌이 잦거나 절대 중복되면 안 되는 경우 고려한다.

## 6. 데이터 접근

### ORM 사용 시 주의점

- N+1 쿼리.
- lazy loading이 API 직렬화 시점에 터지는 문제.
- 영속성 컨텍스트와 트랜잭션 경계 혼동.
- 대량 처리에서 1건씩 flush하는 문제.
- 엔티티를 곧바로 API 응답으로 노출하는 문제.

### SQL 감각

- 인덱스는 읽기를 빠르게 하지만 쓰기 비용과 저장 공간을 늘린다.
- `WHERE`, `JOIN`, `ORDER BY`가 인덱스를 타는지 실행 계획으로 확인한다.
- OFFSET pagination은 뒤 페이지로 갈수록 느려질 수 있다.
- keyset pagination은 무한 스크롤과 대량 목록에 유리하다.

## 7. 비동기와 메시징

메시지 큐를 쓰는 이유:

- 느린 작업을 요청 경로에서 분리한다.
- 트래픽 피크를 버퍼링한다.
- 서비스 간 결합을 낮춘다.
- 재시도와 보상 처리를 체계화한다.

주의점:

- 메시지는 중복 전달될 수 있다고 가정한다.
- 순서가 필요한 키를 명확히 한다.
- poison message는 격리 큐(DLQ)로 보낸다.
- outbox 패턴으로 DB 변경과 메시지 발행의 원자성 문제를 줄인다.

## 8. 외부 연동

모든 외부 연동에는 다음이 필요합니다.

- 연결 timeout, read timeout.
- 재시도 가능 오류와 불가능 오류 구분.
- circuit breaker 또는 bulkhead.
- rate limit 대응.
- 계약 테스트 또는 샌드박스 테스트.
- 장애 시 fallback 또는 사용자 안내.

## 9. 테스트 전략

| 테스트 | 목적 |
|---|---|
| Unit | 도메인 규칙을 빠르게 검증 |
| Integration | DB/큐/외부 클라이언트 설정 검증 |
| Contract | API 제공자·소비자 계약 검증 |
| E2E | 핵심 사용자 흐름 검증 |
| Load | 성능 한계와 병목 확인 |

좋은 테스트는 구현 세부보다 **불변 조건**을 보호합니다.

## 10. 운영 가능한 백엔드

- 모든 요청에 trace id를 전파한다.
- 비즈니스 이벤트와 시스템 이벤트를 구분해 로그를 남긴다.
- p50/p95/p99 지연 시간을 본다.
- 큐 적체량, DB 커넥션 풀, 스레드 풀, GC를 모니터링한다.
- 배포 후 롤백 가능한 스키마 변경을 설계한다.

## 11. 백엔드 체크리스트

- [ ] API 스키마와 오류 형식이 일관된다.
- [ ] 트랜잭션 경계가 도메인 불변 조건과 맞다.
- [ ] 모든 외부 호출에 timeout과 재시도 정책이 있다.
- [ ] 중복 요청·재시도에 멱등하게 동작한다.
- [ ] N+1과 느린 쿼리를 실행 계획으로 확인했다.
- [ ] 인증·인가가 서버에서 강제된다.
- [ ] 로그만으로 장애 흐름을 추적할 수 있다.
