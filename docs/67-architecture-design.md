# 67. 시스템 설계와 아키텍처 종합

> 시스템 설계는 기술 나열이 아니라 요구사항, 제약, 실패 모드, 운영 비용 사이의 균형을 선택하는 일입니다.

## 1. 설계 질문 순서

1. 사용자는 누구이고 핵심 흐름은 무엇인가?
2. 읽기/쓰기 트래픽 규모와 피크는 어느 정도인가?
3. 일관성이 중요한 데이터는 무엇인가?
4. 지연 시간 목표와 가용성 목표는 무엇인가?
5. 장애 시 어떤 기능을 포기할 수 있는가?
6. 보안·규제·감사 요구사항은 무엇인가?
7. 운영팀이 감당할 수 있는 복잡도인가?

## 2. 품질 속성

| 속성 | 질문 |
|---|---|
| 가용성 | 언제나 응답해야 하는가? |
| 일관성 | 오래된 데이터를 허용할 수 있는가? |
| 확장성 | 트래픽 증가를 어떻게 흡수하는가? |
| 성능 | p95/p99 목표가 무엇인가? |
| 보안 | 누가 무엇에 접근할 수 있는가? |
| 관측성 | 장애를 얼마나 빨리 이해할 수 있는가? |
| 변경 용이성 | 요구사항 변경을 어디까지 흡수하는가? |

## 3. 모놀리스와 마이크로서비스

### 모놀리스가 좋은 경우

- 팀이 작다.
- 도메인 경계가 아직 불명확하다.
- 강한 트랜잭션 일관성이 필요하다.
- 운영 역량이 제한적이다.

### 마이크로서비스가 좋은 경우

- 팀과 배포 경계가 분명하다.
- 서비스별 확장 요구가 다르다.
- 장애 격리가 중요하다.
- 조직이 분산 운영 복잡도를 감당할 수 있다.

마이크로서비스는 성능 해결책이 아니라 조직·배포·소유권 분리 전략입니다.

## 4. 동기와 비동기

| 방식 | 장점 | 단점 |
|---|---|---|
| 동기 HTTP/RPC | 단순, 즉시 결과 | 호출 체인 장애 전파 |
| 비동기 메시지 | 버퍼링, 느슨한 결합 | 최종 일관성, 중복, 순서 문제 |
| 배치 | 대량 처리 효율 | 실시간성 낮음 |
| 스트리밍 | 실시간 처리 | 운영 복잡도 |

## 5. 일관성 패턴

- **Strong consistency**: 읽으면 최신 데이터가 보장된다.
- **Eventual consistency**: 시간이 지나면 일관된다.
- **Saga**: 여러 로컬 트랜잭션과 보상 작업으로 전체 흐름을 관리한다.
- **Outbox**: DB 변경과 이벤트 발행 사이의 원자성 문제를 줄인다.
- **Idempotency**: 재시도·중복 호출에도 같은 결과를 만든다.

## 6. 확장 패턴

- Read replica.
- Cache aside.
- Queue buffering.
- Sharding/partitioning.
- CQRS.
- CDN.
- 비동기 precomputation.

확장은 문제를 다른 곳으로 옮깁니다. 캐시는 일관성 문제, 샤딩은 쿼리와 운영 문제, 큐는 지연과 중복 문제를 만듭니다.

## 7. 장애 설계

장애는 예외가 아니라 정상 상황입니다.

- timeout.
- retry with backoff.
- circuit breaker.
- bulkhead.
- graceful degradation.
- fallback.
- rate limiting.
- load shedding.

재시도 폭풍을 막기 위해 클라이언트, 서버, 큐 소비자의 재시도 정책을 함께 봐야 합니다.

## 8. 데이터 설계와 API 설계 연결

- API 응답 요구가 DB 모델을 그대로 노출하게 만들면 장기적으로 변경이 어렵다.
- 읽기 모델과 쓰기 모델을 분리하면 성능과 표현력은 좋아지지만 동기화 책임이 생긴다.
- 페이지네이션, 정렬, 검색 조건은 인덱스와 함께 설계해야 한다.
- 이벤트 스키마는 API보다 더 오래 남을 수 있으므로 버전 관리가 중요하다.

## 9. 설계 문서 템플릿

```md
# 제목

## 배경
왜 이 변경이 필요한가?

## 목표 / 비목표
무엇을 해결하고, 무엇은 이번에 하지 않는가?

## 요구사항
기능, 성능, 보안, 운영, 규제 요구사항.

## 대안
각 대안의 장점, 단점, 비용, 위험.

## 결정
선택한 구조와 이유.

## 장애/운영 계획
모니터링, 알림, 롤백, 마이그레이션, 런북.

## 검증 계획
테스트, 부하 검증, 보안 검토.
```

## 10. 아키텍처 체크리스트

- [ ] 기능 요구사항과 품질 속성이 분리되어 있다.
- [ ] 트래픽 규모와 데이터 증가량 가정이 명시되어 있다.
- [ ] 일관성 수준이 데이터별로 정리되어 있다.
- [ ] 실패 모드와 복구 전략이 있다.
- [ ] 마이그레이션과 롤백 경로가 있다.
- [ ] 운영자가 볼 로그·메트릭·트레이스가 설계되어 있다.
- [ ] 선택하지 않은 대안과 이유가 기록되어 있다.
