# 77. 소프트웨어 엔지니어링 협업과 유지보수

> 개발 실력은 코드 작성만이 아니라 요구사항을 정리하고, 변경을 안전하게 리뷰·배포·운영하는 능력까지 포함합니다.

## 1. 요구사항 읽기

요구사항을 받으면 바로 구현하지 말고 다음을 확인합니다.

- 목표: 왜 하는가?
- 사용자: 누가 쓰는가?
- 성공 기준: 어떻게 완료를 판단하는가?
- 비목표: 이번에 하지 않는 것은 무엇인가?
- 제약: 일정, 보안, 규제, 호환성.
- 위험: 실패하면 무엇이 깨지는가?

## 2. 설계 문서

작은 변경은 PR 설명으로 충분하지만, 큰 변경은 설계 문서가 필요합니다.

포함할 것:

- 배경.
- 목표/비목표.
- 현재 구조.
- 제안 구조.
- 대안과 trade-off.
- 마이그레이션.
- 테스트와 롤백.
- 운영 영향.

## 3. Git 기본기

- 작은 커밋.
- 의도가 드러나는 메시지.
- 관련 없는 변경 분리.
- rebase/merge 전략 이해.
- conflict를 의미 단위로 해결.
- generated file과 source file 구분.

## 4. 코드 리뷰

리뷰어가 볼 것:

- 요구사항을 만족하는가?
- 더 단순한 방법이 있는가?
- edge case가 있는가?
- 보안/성능/운영 문제가 있는가?
- 테스트가 적절한가?
- 이름과 구조가 장기 유지보수에 좋은가?

작성자가 할 것:

- PR 목적과 검증 방법을 적는다.
- 큰 변경은 파일별 설명을 단다.
- 방어적 태도보다 의도 설명에 집중한다.
- 리뷰 반영 후 재검증한다.

## 5. 리팩터링

리팩터링은 기능 변경 없이 구조를 개선하는 일입니다.

원칙:

- 테스트로 현재 동작을 잠근다.
- 작은 단계로 나눈다.
- 이름 변경과 로직 변경을 분리한다.
- 삭제가 추가보다 낫다.
- 추상화는 중복보다 변경 이유를 기준으로 만든다.

## 6. 기술 부채

기술 부채는 나쁜 코드가 아니라 미래 속도를 빌려 현재 속도를 얻은 결정입니다.

관리 방법:

- 부채의 이자를 설명한다.
- 언제 갚을지 조건을 정한다.
- 임시 코드에 만료일과 제거 조건을 둔다.
- 반복 장애나 개발 지연으로 비용을 측정한다.

## 7. 문서화

좋은 문서는 코드가 말하지 못하는 것을 설명합니다.

- 왜 이런 결정을 했는가?
- 어떤 대안을 버렸는가?
- 운영자가 장애 때 무엇을 해야 하는가?
- 새 개발자가 어디서 시작해야 하는가?

문서는 길이보다 최신성과 위치가 중요합니다.

## 8. 릴리즈 노트

릴리즈 노트에는 다음이 필요합니다.

- 사용자 영향.
- breaking change.
- migration guide.
- rollback 방법.
- feature flag 상태.
- 모니터링 포인트.

## 9. 유지보수 가능한 코드

- 이름이 도메인 언어와 맞다.
- 함수가 한 가지 이유로 변경된다.
- side effect가 격리되어 있다.
- 오류 처리가 명시적이다.
- 테스트가 사용 예시 역할을 한다.
- 설정과 비밀값이 코드에서 분리되어 있다.

## 10. 체크리스트

- [ ] 요구사항의 목표와 비목표가 명확하다.
- [ ] PR이 작고 검증 방법이 적혀 있다.
- [ ] 리뷰에서 보안/성능/운영 관점을 확인한다.
- [ ] 리팩터링은 테스트로 보호된다.
- [ ] 기술 부채에 제거 조건이 있다.
- [ ] 운영 런북과 설계 결정이 문서화되어 있다.
