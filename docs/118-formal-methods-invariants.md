# 118. 불변조건, 정형기법, 안전한 상태 설계

> 복잡한 시스템은 “좋은 테스트”만으로 부족할 수 있습니다. 불변조건을 명시하고 상태 공간을 줄여야 합니다.

## 1. 불변조건

불변조건은 시스템이 항상 지켜야 하는 규칙입니다.

예:

- 잔고는 음수가 될 수 없다.
- 체결 수량은 주문 수량을 초과할 수 없다.
- 한 사용자는 다른 tenant 데이터를 볼 수 없다.
- 결제 완료 이벤트는 결제 요청 이후에만 발생한다.

좋은 설계는 불변조건을 코드, DB constraint, 테스트로 중복 보호합니다.

## 2. 상태 머신

상태 전이는 명시적으로 설계합니다.

```text
Created → Pending → Completed
        ↘ Failed → Retried
```

허용되지 않는 전이는 컴파일 타임이나 런타임 검증으로 막습니다.

## 3. Property-Based Testing

예시 기반 테스트는 몇 가지 사례만 봅니다. Property-based testing은 입력을 생성해 성질을 검증합니다.

- 정렬 결과는 항상 오름차순.
- 입금 후 출금 합계는 잔고 변화와 일치.
- serialize 후 deserialize하면 원래 값.
- 같은 idempotency key 요청은 같은 결과.

## 4. Model Checking

TLA+, Alloy 같은 도구는 상태 전이와 동시성 시나리오를 탐색합니다.

적합한 곳:

- 분산 락.
- 리더 선출.
- 결제/정산 상태.
- 재시도와 보상 트랜잭션.
- 권한 정책.

## 5. 타입으로 상태 표현

- nullable 대신 sealed state.
- primitive obsession 줄이기.
- 금액/통화 전용 타입.
- 권한 수준 타입.
- validated input 타입.

잘못된 상태를 표현할 수 없게 만드는 것이 가장 강력한 방어입니다.

## 6. 체크리스트

- [ ] 핵심 불변조건이 문서화되어 있다.
- [ ] 상태 전이가 명시적으로 표현된다.
- [ ] DB constraint와 도메인 검증이 함께 있다.
- [ ] property-based test 후보를 식별했다.
- [ ] 복잡한 동시성/분산 규칙은 모델링을 고려했다.
- [ ] primitive 대신 도메인 타입을 사용한다.
