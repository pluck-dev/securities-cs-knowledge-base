# 50. 백오피스·정산·대사 심화 용어집

증권사 백오피스(정산·대사·회계) 개발자가 업무 코드를 작성할 때 반드시 알아야 할 도메인 용어를 5개 섹션으로 집대성한 문서다. 각 용어마다 실제 개발 시 유의해야 할 포인트를 함께 기재하여, 요건 분석에서 시스템 구현까지 빠른 온보딩을 지원한다.

---

## 정산·결제 업무

| 용어 | 영문/약어 | 뜻 | 개발 포인트 |
|---|---|---|---|
| 일일정산 | Daily Settlement | 매 영업일 종료 후 당일 체결된 모든 거래를 대금·증권 단위로 확정·집계하는 프로세스 | 체결일(T) 기준으로 결제예정일(T+2 등)까지의 미결제 레코드를 별도 테이블로 관리. 날짜 기준을 영업일(bizdate)로 통일 |
| 마감정산 | Final Settlement | 일일정산 완료 후 수정이 불가한 확정 상태로 전환하는 최종 정산 단계 | 마감 플래그(settled_flag) 설정 시 이후 DML 차단 트리거 또는 애플리케이션 레벨 제어 필요 |
| 결제내역 | Settlement Detail | 결제일에 실제로 이루어진 자금·증권 이동 내역 | 체결번호(trade_id), 결제번호(settlement_id), 결제금액, 결제수량 등 키 컬럼을 분리 관리 |
| 결제대사 | Settlement Reconciliation | 사내 결제 기록과 외부 기관(예탁원·청산소 등) 기록을 비교하여 일치 여부를 확인하는 작업 | 외부 전문(FTP/SFTP 수신) 파싱 후 내부 DB와 키 매칭. 불일치 건은 recon_diff 테이블로 격리 |
| 매매대사 | Trade Reconciliation | 체결 시스템(주문·체결)과 백오피스 원장 간 체결 수량·금액이 일치하는지 확인하는 절차 | T+0 배치에서 체결 피드(FIX/FTP)와 원장 INSERT 결과를 건별 매칭. 차이 발생 시 알림 발송 |
| 잔고대사 | Position Reconciliation | 사내 보유 잔고와 예탁원·청산원 잔고를 비교하여 정확성을 검증하는 절차 | 종목별(isin), 계좌별 잔고를 피드와 JOIN 후 sum 비교. 허용 오차 기준 내부 정책 확인 필요 |
| 예탁대사 | Depository Reconciliation | 예탁결제원 보관 잔고와 당사 예탁원장 간 대사. 잔고대사의 예탁원 특화 버전 | 예탁원 SAFE+ 전문(파일) 수신 주기(T+1 오전)에 맞춰 배치 타이밍 설계 |
| 차감결제 | Netting Settlement | 동일 결제일에 발생한 다수 거래의 매도·매수 금액/수량을 상계 후 차액만 이동시키는 방식 | 종목·계좌·결제일 기준으로 net_qty, net_amount 집계. 건별(gross) 결제와 플래그로 구분 |
| 다자간차감 | Multilateral Netting | 청산기관(CCP)이 참여 회원사 전체의 매수·매도를 상계하여 최소 결제 포지션만 산출하는 방식 | CCP로부터 수신하는 최종 포지션 파일 기준으로 결제 처리. 사내 건별 체결과 별도 조회 경로 필요 |
| 결제이행 | Settlement Completion | 결제일에 약정된 대금·증권이 정상적으로 이전 완료된 상태 | settled_at 타임스탬프 기록. 이후 취소·정정 불가 처리 로직 추가 |
| 결제불이행 | Settlement Failure / Default | 결제일까지 대금 또는 증권을 이전하지 못한 상태. 연체료·담보 처분 대상 | status='FAILED' 시 알림, 연체이자 자동 계산 배치(예시·사내기준 확인), 담보 처분 트리거 연계 |
| 결제유보 | Settlement Suspension | 특정 사유(분쟁, 법원명령, AML 등)로 결제를 일시 보류하는 상태 | 유보 사유 코드(suspend_reason_cd), 유보 해제일(release_dt) 컬럼 설계. 감사 로그 필수 |
| 결제분리 | Settlement Segregation | 고객 자산과 회사 고유 자산의 결제 경로를 물리적으로 분리하는 규정 | 계좌 유형(acc_type: 고유/위탁) 기준으로 결제 테이블 혹은 파티션 분리 |
| 미수 | Margin Call / Receivable | 고객이 매수 후 결제대금을 납입하지 못한 상태. 증거금 부족 시 반대매매 대상 | 미수 발생일(margin_call_dt), 반대매매 예정일(forced_sell_dt) 관리. 예시·사내기준 확인 |
| 미지급 | Payable | 매도 후 증권을 결제일까지 납부하지 못한 상태 | 잔고 부족 시 차입결제(SBL) 연계 또는 불이행 처리 분기 설계 |
| 청산기관 | CCP (Central Counterparty Clearing) | 매도자와 매수자 사이에 개입하여 결제 이행을 보증하는 기관(예: 한국거래소 청산소) | CCP 제공 API/파일 기준으로 마진·담보 산출 로직 연계. 회원 자격·한도 정보 별도 관리 |
| DVP | Delivery Versus Payment | 증권 인도와 대금 지급을 동시에 처리하여 결제 리스크를 최소화하는 원칙 | 증권 이전과 자금 이체를 하나의 트랜잭션(또는 2PC)으로 묶거나, 양쪽 완료 확인 후 커밋 |
| 결제지시 | Settlement Instruction | 예탁원·은행 등 외부 기관에 자금 또는 증권 이전을 지시하는 전문(메시지) | ISO 15022/20022 형식 준수. 전문 생성→전송→수신확인(ACK) 상태 기계 구현 |
| 결제계좌 | Settlement Account | 결제 대금이 입출금되는 은행 계좌 또는 예탁원 내 자금계정 | 계좌번호 암호화(AES-256), 마스킹 출력, 접근 권한 분리 필수 |
| 자금이체 | Fund Transfer | 결제 대금을 은행 계좌 간 이동시키는 행위 | 이체 요청→처리중→완료/실패 상태 관리. 멱등성 키(idempotency key) 설계 필수 |
| 한은금융망 | BOK-Wire+ | 한국은행이 운영하는 거액 자금결제 시스템. 증권 대금결제에 활용 | 전문 규격(BOK-Wire+ 메시지 표준) 숙지. 운영 시간(예시·사내기준 확인) 내 배치 설계 |
| 증권결제 | Securities Settlement | 매매 계약에 따라 증권을 인도·수령하는 프로세스 | 예탁원 전자등록 기반. 실물 증권 잔존 시 별도 처리 경로 유지 |
| 대금결제 | Cash Settlement | 매매 계약에 따라 현금(대금)을 지급·수령하는 프로세스 | 원화와 외화(USD 등) 결제 통화를 currency_cd로 구분. 환율 적용 시점 기준 명확화 |
| 결제주기 | Settlement Cycle (T+N) | 체결일(T)로부터 결제일까지의 영업일 수. 국내 주식 T+2, 일부 상품 T+1 또는 T+0 | 결제일 = add_bizdays(trade_date, N). 공휴일 캘린더 테이블 기반 계산 로직 필수 |
| 결제통화 | Settlement Currency | 결제가 이루어지는 통화 단위 | 다통화 지원 시 통화별 계좌, 환율 스냅샷 테이블 설계 |

## 회계·원장 처리

| 용어 | 영문/약어 | 뜻 | 개발 포인트 |
|---|---|---|---|
| 계정과목 | Account Title / Chart of Accounts | 거래를 분류하는 회계 항목(예: 현금, 유가증권, 위탁매매수익 등) | 계정코드(account_cd) 체계를 표준화. 변경 이력 관리(유효기간 컬럼) 필요 |
| 분개 | Journal Entry | 거래를 차변(Debit)과 대변(Credit)으로 나누어 기록하는 회계 행위 | journal_entry 테이블: 헤더(일자·전표번호)와 라인(계정·금액·차대구분) 1:N 구조 |
| 차변 | Debit (Dr.) | 분개에서 자산 증가 또는 부채·자본 감소를 기록하는 왼쪽 항목 | dr_amount 컬럼으로 관리. 차대 합계 일치 검증 로직 필수 |
| 대변 | Credit (Cr.) | 분개에서 부채·자본 증가 또는 자산 감소를 기록하는 오른쪽 항목 | cr_amount 컬럼. 전표 저장 시 SUM(dr) = SUM(cr) 체크 |
| 복식부기 | Double-Entry Bookkeeping | 모든 거래를 차변과 대변에 동시에 기록하는 회계 원칙 | 분개 생성 서비스에서 항상 대차 균형을 강제 검증. 불균형 전표 저장 차단 |
| 시산표 | Trial Balance | 특정 시점까지의 모든 계정 잔액을 차변/대변으로 나열하여 대차 균형을 검증하는 표 | 회계 마감 배치 후 자동 생성. 불균형 시 마감 중단 및 알림 |
| 일계표 | Daily Ledger Summary | 당일 발생한 전체 전표를 계정과목별로 집계한 일별 요약표 | 일마감 배치 최종 단계에서 생성. 검증용 체크섬 컬럼 추가 권장 |
| 월계표 | Monthly Ledger Summary | 당월 발생 전표를 계정과목별로 집계한 월별 요약표 | 월마감 배치 결과물. 전월 잔액 이월(carry-forward) 로직 구현 필요 |
| 총계정원장 | General Ledger (GL) | 모든 계정과목의 거래 내역과 잔액을 기록하는 핵심 회계 장부 | GL 테이블은 불변(immutable) 원칙. 수정 시 역분개(reversing entry) 생성 |
| 보조원장 | Subsidiary Ledger | 총계정원장의 특정 계정을 세부 항목별로 기록하는 보조 장부(예: 고객별 예탁금 원장) | GL과 보조원장 합계를 주기적으로 대사하는 배치 설계 필요 |
| 미결제잔액 | Outstanding Balance | 아직 결제가 완료되지 않은 미확정 잔액 | 결제일(settlement_date) 기준으로 T+N 이전 건을 별도 조회. 마감 시 자동 확정 처리 |
| 가수금 | Suspense Receipt | 입금 원인이 불분명하여 임시로 기록하는 임시 계정 | 가수금 계정(account_cd: 특정 코드) 발생 시 담당자 알림. 해소 기한(예시·사내기준 확인) 관리 |
| 가지급금 | Suspense Payment | 지급 원인이 불분명하거나 정산 전 임시로 지급한 금액을 기록하는 임시 계정 | 가지급금 잔액 조회 리포트 자동화. 미해소 기간 경과 시 에스컬레이션 |
| 충당금 | Allowance / Provision | 미래 손실(대손, 미수금 등)에 대비하여 당기에 미리 비용으로 인식하는 금액 | 충당금 설정·환입 분개 자동화. 설정 기준율(예시·사내기준 확인) 파라미터화 |
| 손익계정 | Profit and Loss Account (P&L) | 수익과 비용을 집계하여 당기 손익을 산출하는 계정 그룹 | 결산 배치에서 손익 계정 잔액을 이익잉여금으로 대체하는 마감 분개 자동 생성 |
| 자기매매 | Proprietary Trading | 증권사가 자기 고유 자산으로 매매하는 행위 | acc_type='P'(proprietary) 구분. 고유재산 원장과 투자자재산 원장 철저히 분리 |
| 위탁매매 | Brokerage Trading | 고객 주문을 대리 집행하는 매매 행위 | acc_type='C'(client). 수수료 발생, 예탁금 차감 로직 연계 |
| 고유재산 | Proprietary Assets | 증권사 자신의 자산. 투자자 예탁자산과 혼용 금지 | DB 레벨에서 고유/위탁 스키마 또는 파티션 분리. 교차 집계 쿼리 방지 |
| 투자자재산 | Client Assets | 고객이 증권사에 예탁한 자산. 고유재산과 명확히 분리 보관 | 투자자보호기금 관련 별도예치 금액 집계 자동화 |
| 예탁금별도예치 | Segregated Deposit of Customer Funds | 고객 예탁금을 증권사 고유재산과 분리하여 외부 기관(은행·예탁원)에 별도 보관하는 의무 | 별도예치 대상 금액 계산 배치(매일 실행). 예치 부족 시 즉시 알림. 예시·사내기준 확인 |
| 회계마감 | Accounting Close | 특정 기간(일·월·분기·연) 회계 처리를 종료하고 장부를 확정하는 절차 | 마감 상태 테이블(period_close): 기간코드, 마감유형, 마감자, 마감일시. 마감 후 전표 입력 차단 |
| 결산 | Financial Settlement / Closing | 회계 연도 말 재무제표를 확정하는 연간 마감 절차 | 결산 배치는 월마감 배치를 12회 순차 실행 후 손익 대체 분개 추가. 감사 대응 로그 보존 |
| 역분개 | Reversing Entry | 오류 수정이나 임시 계상 항목을 취소하기 위해 원래 분개의 차·대를 바꾸어 입력하는 전표 | 원전표 ID(original_journal_id) 참조. 역분개 생성 시 원전표 상태를 'REVERSED'로 갱신 |
| 전표 | Voucher | 거래 사실을 증명하고 분개를 기록하는 회계 문서 단위 | 전표번호 채번 규칙(년+월+일련번호) 표준화. 전표 유형(입금/출금/대체) 코드 관리 |
| 이월 | Carryforward | 전기 잔액을 당기 초 잔액으로 넘기는 절차 | 연·월 마감 후 잔액 이월 배치. 이월 전·후 잔액 대사 검증 필수 |

## 수수료·비용 정산

| 용어 | 영문/약어 | 뜻 | 개발 포인트 |
|---|---|---|---|
| 위탁수수료정산 | Brokerage Commission Settlement | 고객 매매 체결에 따라 발생하는 위탁수수료를 집계·징수·정산하는 프로세스 | 수수료율 테이블(fee_rate): 계좌유형·채널·구간별 다단계 설계. 체결금액 × 요율 계산 |
| 유관기관수수료 | Exchange/Depository Fees | 한국거래소·예탁결제원·금융투자협회 등에 납부하는 수수료 | 기관별 수수료율(예시·사내기준 확인)·납부 주기 파라미터 관리. 자동 납부 배치 연계 |

|---|---|---|---|
| 온라인마감 | Online Close | 실시간 거래 시스템을 종료하고 장 마감 상태로 전환하는 절차 | 마감 신호 수신 후 신규 주문 차단 → 잔여 체결 처리 완료 → 배치 트리거 순서 정의 |
| 배치마감 | Batch Close | 장 마감 후 정산·대사·분개·보고서 생성 등 일괄 처리 배치군 전체를 완료하는 과정 | DAG(Directed Acyclic Graph) 또는 배치 의존성 그래프 설계. 실패 시 재시작 지점(checkpoint) 관리 |

| 야간배치 | Overnight Batch | 장 종료 후 익일 개장 전까지 실행되는 대용량 정산·마감 배치군 | 처리 순서: 체결확정 → 대사 → 분개 → 보고서 → 이월. 최대 실행 허용 시간(예시·사내기준 확인) |

| 배치실패알림 | Batch Failure Notification | 배치 실행 중 오류 발생 시 담당자에게 즉시 통보하는 시스템 | 알림 채널(이메일·Slack·SMS) 설정, 에러 코드·스택트레이스 포함, 에스컬레이션 정책(예시·사내기준 확인) |
| 배치의존성 | Batch Dependency | 배치 A 완료 후 배치 B를 실행해야 하는 선행·후행 관계 | DAG 또는 선행배치완료상태(PRE_STATUS) 체크. 순환 의존 방지 검증 로직 |
| 배치이력 | Batch History | 각 배치 실행의 시작·종료 시간, 처리 건수, 성공/실패를 기록한 이력 | batch_exec_log 테이블 설계. 보존 기간(예시·사내기준 확인), 파티션 관리 |
| 야간마감시간 | Overnight Close Window | 야간배치가 반드시 완료되어야 하는 시간 범위(개장 전까지) | SLA 설정(예시·사내기준 확인). 초과 시 자동 알림 및 비상 대응 절차 연계 |

| 이상거래보고 | Suspicious Transaction Report (STR) | AML(자금세탁방지) 규정에 따라 이상 거래를 금융정보분석원(KoFIU)에 신고하는 보고 | 이상거래 탐지 룰 엔진 연계. 신고 기한(예시·사내기준 확인) 자동 관리. 신고 이력 암호화 보존 |
| 고객확인정보보고 | CDD/KYC Report | 고객 신원 확인(KYC) 및 고객 실사(CDD) 결과를 감독 기관에 제출하는 보고 | KYC 갱신 주기(예시·사내기준 확인) 모니터링. 미갱신 고객 거래 제한 자동화 |
| 산출물목록 | Report Catalog | 시스템에서 생성·관리되는 모든 보고서·문서의 목록과 메타데이터 | 보고서 ID, 생성 주기, 담당 배치, 출력 형식, 보존 기간 관리 테이블 설계 |
| 전자금융거래보고 | Electronic Financial Transaction Report | 전자금융거래법에 따라 이상·대량 거래를 금감원에 보고하는 자료 | 임계금액(예시·사내기준 확인) 초과 거래 자동 감지·보고. 보고 기한 SLA 관리 |
| 보고서생성배치 | Report Generation Batch | 정해진 주기에 자동으로 보고서를 생성하는 배치 | 보고서별 생성 시간 모니터링. 생성 실패 시 재시도 및 알림. 대용량 생성 시 비동기 처리 |
