# 66. 보안 엔지니어링 기본

> 보안은 기능이 아니라 시스템의 기본 속성입니다. 신뢰할 수 없는 입력, 사용자, 네트워크, 의존성을 전제로 설계해야 합니다.

## 1. 보안 사고방식

- 기본 거부, 필요한 것만 허용한다.
- 최소 권한을 부여한다.
- 입력은 항상 검증한다.
- 비밀값은 코드·로그·이미지에 두지 않는다.
- 실패해도 안전한 방향으로 동작하게 한다.
- 보안 통제는 우회하기 어렵고 관측 가능해야 한다.

## 2. 인증(Authentication)

- 비밀번호는 평문 저장 금지. 강한 해시 알고리즘과 salt를 사용한다.
- MFA는 계정 탈취 위험을 줄인다.
- 세션 탈취를 고려해 만료, 회전, 폐기 정책을 둔다.
- OAuth2/OIDC는 인증 위임과 identity 정보를 다룬다.

## 3. 인가(Authorization)

권한 모델:

- RBAC: 역할 기반.
- ABAC: 속성 기반.
- ACL: 리소스별 접근 목록.
- Policy-based: 정책 엔진으로 판단.

실무 원칙:

- 서버에서 최종 검증한다.
- 목록 조회와 상세 조회 모두 권한 필터가 필요하다.
- 관리자 기능은 감사 로그를 남긴다.
- multi-tenant 시스템은 tenant boundary를 모든 쿼리에 반영한다.

## 4. 암호화

- 전송 중 암호화: TLS.
- 저장 시 암호화: DB, 디스크, object storage.
- 애플리케이션 레벨 암호화: 특히 민감한 필드.
- 키 관리는 KMS/HSM 같은 별도 체계로 한다.

주의: 암호화는 키 관리가 절반입니다. 키가 로그나 코드에 있으면 암호화의 의미가 약해집니다.

## 5. 웹 보안

| 취약점 | 방어 |
|---|---|
| XSS | 출력 인코딩, 안전한 템플릿, CSP |
| CSRF | SameSite, CSRF token, Origin 검증 |
| SQL Injection | prepared statement, ORM 바인딩 |
| SSRF | allowlist, metadata endpoint 차단 |
| Path Traversal | 경로 정규화, allowlist |
| Open Redirect | redirect 대상 allowlist |
| Clickjacking | frame-ancestors, X-Frame-Options |

## 6. API 보안

- rate limit과 abuse detection.
- 요청 크기 제한.
- 스키마 검증.
- 인증 실패와 권한 실패 구분.
- 민감 오류 메시지 숨김.
- idempotency key와 replay 방지.

## 7. Secret 관리

- `.env`는 로컬 개발용이며 저장소에 커밋하지 않는다.
- 운영 secret은 secret manager에서 주입한다.
- secret rotation 절차가 있어야 한다.
- 로그 마스킹과 사고 시 폐기 절차를 준비한다.

## 8. 공급망 보안

- lockfile을 커밋한다.
- 의존성 취약점 스캔을 수행한다.
- 불필요한 dependency를 줄인다.
- 빌드 산출물의 출처와 무결성을 확인한다.
- CI 토큰 권한을 최소화한다.

## 9. 개인정보와 감사

- 수집 목적과 보관 기간을 명확히 한다.
- 민감정보 접근은 감사 로그를 남긴다.
- 개발·테스트 환경에는 실제 개인정보를 넣지 않는다.
- 마스킹, 익명화, 가명화를 구분한다.

## 10. 보안 체크리스트

- [ ] 모든 입력은 서버에서 검증한다.
- [ ] 인증과 인가가 분리되어 구현되었다.
- [ ] secret이 저장소·이미지·로그에 없다.
- [ ] SQL/명령/경로 주입 방어가 있다.
- [ ] 관리자와 민감 데이터 접근 로그가 남는다.
- [ ] 의존성 취약점과 라이선스를 점검한다.
- [ ] 사고 시 키 폐기와 토큰 무효화 절차가 있다.
