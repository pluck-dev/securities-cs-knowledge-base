# 01. 큰 그림: 증권사에서 개발자가 뭘 만드나?

증권사 시스템은 크게 다섯 영역으로 나뉩니다. 어느 팀에 배치되든 이 지도가 머릿속에 있어야 합니다.

| 영역 | 하는 일 | 예시 |
|------|---------|------|
| **MTS/HTS** | 사용자가 보는 매매 화면 | 키움 영웅문, 토스증권 앱 |
| **주문/체결(트레이딩)** | 주문을 받아 거래소로 보내고 체결 결과 처리 | "삼성전자 10주 매수" → 거래소 |
| **시세(시장데이터)** | 실시간 가격, 호가, 차트 데이터 | 실시간 현재가 스트리밍 |
| **계좌/원장(원장계)** | 잔고, 예수금, 손익 계산 | "내 계좌에 몇 주, 얼마" |
| **백오피스** | 정산, 세금, 리포트, 컴플라이언스 | 양도세, 거래내역 |

## 코틀린은 어디서 쓰나?

코틀린은 보통 **백엔드 서버** (주문, 계좌, 시세 중계, API 서버) 쪽에서 많이 씁니다.
- JVM(자바 가상머신) 위에서 돌기 때문에, 기존 자바 시스템과 100% 호환됩니다.
- 그래서 "수십 년 된 자바 시스템 + 신규 코틀린 서비스"가 공존하는 형태가 흔합니다.

## 첫 출근하면 물어볼 질문 2개

방향을 빠르게 잡는 핵심 질문입니다.

1. **"우리 팀 시스템이 위 표 중 어디에 해당하나요?"**
   → 주문계인지, 시세계인지, 원장계인지에 따라 배워야 할 게 달라집니다.

2. **"실시간 처리와 배치 처리 중 뭐가 주력인가요?"**
   - 실시간(real-time): 주문/시세처럼 ms 단위 속도가 중요
   - 배치(batch): 장 마감 후 정산/리포트처럼 대량 일괄 처리

## 팀별 특징 요약

| 팀 | 핵심 역량 | 난이도 포인트 |
|----|----------|--------------|
| 주문/체결 | 저지연, 동시성, 정확성 | 속도와 정합성 동시 달성 |
| 시세 | 대량 스트리밍, 자료구조 | 초당 수천~수만 건 처리 |
| 원장/계좌 | 데이터 정합성, 트랜잭션 | 1원도 틀리면 안 됨 |
| 백오피스 | 배치, 규제 대응 | 세법/규정 변화 추적 |
| MTS/HTS 백엔드 | API 설계, 트래픽 | 9시 트래픽 폭증 대응 |
