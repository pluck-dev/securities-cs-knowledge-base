# 04. 증권 도메인 지식 (핵심)

코드보다 **이 업무 흐름과 용어**를 아는 사람이 회사에서 인정받습니다. 투자 경험이 빛나는 곳입니다.

## 주문 → 체결 생명주기 (가장 중요!)

```
1. 주문 접수    (사용자가 "삼성전자 10주 매수" 누름)
2. 주문 검증    (예수금 충분? 호가 단위 맞나? 시장 열렸나?)
3. 거래소 전송  (한국거래소 KRX로 주문 송신)
4. 체결/미체결  (전부체결 / 일부체결 / 미체결)
5. 결과 통보    (사용자에게 체결 알림)
6. 잔고/예수금 반영 (원장 업데이트)
7. 결제(T+2)    (실제 돈/주식 이동은 2영업일 뒤)
```

개발자는 이 단계마다 데이터가 정확히 흘러가는지를 책임집니다.
**"돈과 관련된 숫자는 1원도 틀리면 안 된다"**가 철칙입니다.

## 꼭 알아야 할 주문 유형

| 유형 | 뜻 |
|------|-----|
| **지정가 (LIMIT)** | 가격을 정해서 주문 ("7만원에 매수") |
| **시장가 (MARKET)** | 현재 시세로 즉시 체결 ("지금 당장 매수") |
| **조건부지정가 / IOC / FOK** | 체결 조건이 붙은 고급 주문 |

## 체결 유형

| 유형 | 뜻 | 개발 난이도 |
|------|-----|------------|
| **전부체결** | 주문 수량 전부 거래됨 | 쉬움 |
| **일부체결(부분체결)** | 일부만 거래되고 나머지는 대기 | 까다로움 (잔량 추적) |
| **미체결** | 아직 거래 안 됨 | 정정/취소 처리 필요 |

## 핵심 용어 사전 (개발자 버전)

| 용어 | 뜻 | 개발 포인트 |
|------|-----|------------|
| **호가(Quote)** | 사겠다/팔겠다 가격 목록 | 실시간 데이터, 초당 수천 건 |
| **호가단위(Tick)** | 가격이 움직이는 최소 단위 (예: 5만원↑은 50원) | 주문 검증 로직 필수 |
| **체결(Execution/Fill)** | 주문이 실제 거래됨 | 부분체결 처리가 까다로움 |
| **예수금** | 계좌의 현금 | 주문 가능 금액 계산의 기준 |
| **증거금** | 주문에 필요한 최소 보증금 | 신용/미수 거래에서 중요 |
| **잔고(Position)** | 보유 종목/수량 | 평단가 계산 로직 |
| **평단가** | 매수 평균 단가 | 계산 실수 잦은 부분 |
| **T+2 결제** | 거래 2영업일 후 정산 | 영업일 계산 로직 |
| **시장가/지정가** | 주문 유형 | 분기 처리 |
| **장 운영시간** | 09:00~15:30, 동시호가 등 | 시간 기반 검증 |
| **종목코드** | 6자리 (삼성전자 005930) | 데이터 키로 자주 씀 |

> 전체 용어집은 [09-glossary.md](09-glossary.md) 참고.

## 평단가 계산 예시 (실수 잦은 로직)

```
기존: 삼성전자 10주, 평단 70,000원  → 평가금액 700,000원
추가: 5주를 80,000원에 매수          → 매수금액 400,000원

새 평단 = (700,000 + 400,000) / (10 + 5) = 1,100,000 / 15 = 73,333.33...원
```

- 나눗셈에서 소수점이 생기므로 `BigDecimal`의 반올림 정책(scale, RoundingMode)을 명확히 정해야 합니다.
- 매도 시에는 평단가가 바뀌지 않고 수량만 줄어드는 게 보통입니다 (회사 정책 확인).

## 돈을 다루는 개발자의 철칙 ⚠️

1. **금액은 절대 `Double`로 계산하지 않는다.** → `BigDecimal` 사용.
2. **모든 거래는 로그/감사 추적이 남아야 한다** (감독 규제).
3. **동시성**: 같은 계좌에 동시 주문이 들어와도 잔고가 꼬이면 안 됨 (락/트랜잭션).
4. **멱등성(idempotency)**: 같은 주문이 중복 전송돼도 두 번 처리되면 안 됨.
5. **장애 = 금전 사고**. 그래서 테스트와 검증이 다른 업계보다 훨씬 빡셉니다.
