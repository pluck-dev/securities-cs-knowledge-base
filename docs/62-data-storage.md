# 62. 데이터베이스와 저장소

> 데이터 저장은 단순 CRUD가 아니라 정확성, 성능, 복구, 감사 가능성을 함께 설계하는 일입니다.

## 1. 저장소 선택

| 저장소 | 강점 | 적합한 곳 |
|---|---|---|
| RDBMS | 트랜잭션, SQL, 일관성 | 주문, 계좌, 결제, 원장 |
| Document DB | 유연한 스키마 | 이벤트 문서, CMS, 프로필 |
| Key-Value | 단순 빠른 조회 | 세션, 캐시, rate limit |
| Wide Column | 대량 분산 쓰기 | 로그, 시계열 유사 데이터 |
| Search Engine | 전문 검색, 랭킹 | 문서 검색, 로그 검색 |
| Object Storage | 대용량 파일 | 이미지, 백업, 리포트 |

## 2. RDBMS 핵심

### 테이블 설계

- 엔티티의 정체성을 나타내는 primary key가 필요하다.
- 외래키는 데이터 무결성을 지키지만 운영·배포 정책과 함께 결정한다.
- 정규화는 중복과 이상 현상을 줄인다.
- 반정규화는 읽기 성능을 위해 선택하지만 동기화 책임이 생긴다.

### SQL 기본

- `SELECT`는 필요한 컬럼만 조회한다.
- `JOIN`은 관계를 명확히 표현하지만 카디널리티 폭발을 조심한다.
- `GROUP BY`와 `ORDER BY`는 임시 정렬/해시 비용이 클 수 있다.
- `LIMIT/OFFSET`은 뒤 페이지에서 느려질 수 있다.

## 3. 인덱스

인덱스는 책의 색인처럼 원하는 행을 빨리 찾게 해줍니다.

### B-Tree 인덱스가 유리한 조건

- 동등 조건: `where account_id = ?`
- 범위 조건: `where created_at between ? and ?`
- 정렬: `order by created_at desc`
- 복합 조건: `(account_id, created_at)`처럼 선두 컬럼부터 사용할 때

### 주의점

- 인덱스가 많으면 쓰기가 느려진다.
- 낮은 선택도의 컬럼은 단독 인덱스 효과가 작을 수 있다.
- 함수로 컬럼을 감싸면 인덱스를 못 탈 수 있다.
- 실행 계획으로 확인해야 한다.

## 4. 트랜잭션과 ACID

| 속성 | 의미 |
|---|---|
| Atomicity | 모두 성공하거나 모두 실패 |
| Consistency | 불변 조건을 깨지 않음 |
| Isolation | 동시에 실행되어도 간섭을 제어 |
| Durability | 커밋된 데이터는 장애 후에도 보존 |

격리 수준이 낮으면 성능은 좋아질 수 있지만 이상 현상이 생길 수 있습니다. 격리 수준은 DB 기본값을 외우는 것보다 실제 업무 불변 조건과 충돌 패턴을 기준으로 정해야 합니다.

## 5. 락과 동시성 제어

- **낙관적 락**: 버전 컬럼으로 충돌을 감지한다.
- **비관적 락**: 먼저 잠그고 처리한다.
- **분산 락**: 여러 프로세스가 공유 자원을 다룰 때 사용하지만 만능이 아니다.
- **unique constraint**: 중복 방지에는 애플리케이션 체크보다 강력하다.

## 6. NoSQL과 이벤트 저장

NoSQL은 SQL을 몰라도 되는 저장소가 아니라 다른 trade-off를 선택한 저장소입니다.

- 스키마가 없다는 것은 검증 책임이 애플리케이션에 온다는 뜻이다.
- 쿼리 패턴에 맞춰 데이터를 중복 저장할 수 있다.
- 분산 저장소는 일관성과 가용성의 균형이 중요하다.
- 이벤트 저장은 append-only와 재처리에 강하지만 스냅샷·버전 마이그레이션이 필요하다.

## 7. 캐시

### 패턴

| 패턴 | 설명 |
|---|---|
| Cache Aside | 애플리케이션이 캐시 miss 시 DB 조회 후 저장 |
| Read Through | 캐시 계층이 DB 조회를 대신 수행 |
| Write Through | 쓰기 시 캐시와 DB를 함께 갱신 |
| Write Behind | 캐시에 먼저 쓰고 나중에 DB 반영 |

### 위험

- stale data.
- cache stampede.
- hot key.
- 무효화 누락.
- 캐시 장애가 전체 장애로 번지는 현상.

## 8. 백업과 복구

백업은 존재가 아니라 **복구 가능성**이 검증되어야 합니다.

- RPO: 얼마나 많은 데이터 손실을 허용할 수 있는가?
- RTO: 얼마나 빨리 복구해야 하는가?
- PITR: 특정 시점 복구가 가능한가?
- 복구 리허설: 실제로 복원해본 적이 있는가?

## 9. 데이터 마이그레이션

안전한 스키마 변경 순서:

1. 새 컬럼/테이블을 추가한다.
2. 애플리케이션이 양쪽을 읽거나 새 구조를 선택적으로 읽게 한다.
3. 백필을 작은 배치로 수행한다.
4. 쓰기 경로를 새 구조로 전환한다.
5. 모니터링 후 구 구조를 제거한다.

## 10. 저장소 체크리스트

- [ ] 주요 쿼리의 실행 계획을 확인했다.
- [ ] 트랜잭션 경계와 격리 수준이 업무 규칙에 맞다.
- [ ] unique/foreign/check constraint로 중요한 무결성을 보호한다.
- [ ] 캐시 무효화 전략이 있다.
- [ ] 백업 복구를 실제로 테스트했다.
- [ ] 스키마 변경이 롤백 가능한 순서로 설계되었다.
